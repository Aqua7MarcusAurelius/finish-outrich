# Формат истории переписки в промпте

Спецификация формата, в котором история диалога передаётся в LLM
при генерации ответов AutoChat.

Документ описывает **только формат отдачи истории**. Не код, не правила
ожидания обработки медиа, не остальные части промпта.


## Общие правила

1. Каждое сообщение — самостоятельный блок. Между блоками одна пустая
   строка. Внутри блока пустых строк нет.

2. Подписи полей пишутся на английском, с заглавной буквы. Полный
   список допустимых подписей:
   `Author`, `Date`, `Weekday`, `Time`,
   `Text`,
   `Voice`, `Audio`, `Photo`, `Video`, `VideoNote`, `Sticker`, `GIF`,
   `Document`,
   `Description`, `Transcription`.
   Других подписей не бывает.

3. Значения-маркеры пишутся по-русски, с заглавной буквы. К ним
   относятся:
   - автор сообщения: `Ты`, `Собеседник`;
   - день недели: `Понедельник`, `Вторник`, `Среда`, `Четверг`,
     `Пятница`, `Суббота`, `Воскресенье`.

4. Значения-содержимое (текст сообщений и расшифровки медиа) пишутся
   как есть. Регистр и пунктуацию автора не меняем.

5. Технические статусы пишутся английскими словами в круглых скобках
   строчными буквами: `(failed)`, `(empty)`. Статуса `(processing)`
   в истории не бывает: на момент сборки истории все медиа считаются
   либо обработанными (`done`), либо неуспешными (`failed`).

6. Структура верха каждого блока строго фиксирована и идёт в этом
   порядке:
   - `Author`
   - `Date`
   - `Weekday`
   - `Time`
   После — поля содержимого (`Text` и/или вложения).

7. Форматы дат и времени:
   - дата: `ДД.ММ.ГГГГ` (например, `12.12.2026`);
   - время: `ЧЧ:ММ:СС` (например, `12:35:45`);
   - день недели: полным словом по-русски с заглавной.

8. Любое вложение разворачивается с отступом в два пробела и идёт со
   своими подполями. Даже если подполе одно. Единообразие важнее
   компактности — простое фото пишется так:

   ```
   Photo 1:
     Description: рыжий корги на траве
   ```

9. Все вложения нумеруются, всегда. Даже если вложение одно. Нумерация
   идёт внутри типа отдельно: два фото и одно видео — это
   `Photo 1`, `Photo 2`, `Video 1` (а не `Photo 1`, `Photo 2`, `Video 3`).

10. Пустые поля не печатаются. Если у голосового нет транскрипции, у
    видео нет звука, у документа нет описания — соответствующее
    подполе просто отсутствует. Если у вложения вообще не оказалось
    ни одного подполя — само вложение тоже не печатается.


## Соответствие типов

| Тип в БД (`media.type`) | Подпись блока  | Возможные подполя           |
|-------------------------|----------------|-----------------------------|
| (нет media)             | `Text`         | —                           |
| `voice`                 | `Voice`        | `Transcription`             |
| `audio`                 | `Audio`        | `Transcription`             |
| `video_note`            | `VideoNote`    | `Description`, `Transcription` |
| `video`                 | `Video`        | `Description`, `Transcription` |
| `photo`                 | `Photo`        | `Description`               |
| `sticker`               | `Sticker`      | `Description`               |
| `gif`                   | `GIF`          | `Description`               |
| `document`              | `Document`     | `Description`               |


## Каталог по типам сообщений

### Текстовое сообщение

```
Author: Собеседник
Date: 12.12.2026
Weekday: Суббота
Time: 12:35:45
Text: Привет котик!
```

### Своё сообщение

То же самое, что текстовое от собеседника, только `Author: Ты`.

```
Author: Ты
Date: 12.12.2026
Weekday: Суббота
Time: 12:36:04
Text: Привет
```

### Пустое сообщение

Редкий кейс: например, стикер удалили в Telegram, остался только каркас
сообщения без содержимого.

```
Author: Собеседник
Date: 12.12.2026
Weekday: Суббота
Time: 12:42:00
Text: (empty)
```

### Голосовое

```
Author: Собеседник
Date: 12.12.2026
Weekday: Суббота
Time: 12:36:20
Voice 1:
  Transcription: а как тебя зовут?
```

### Голосовое — транскрипция не удалась

```
Author: Собеседник
Date: 12.12.2026
Weekday: Суббота
Time: 12:36:20
Voice 1:
  Transcription: (failed)
```

### Аудиофайл

```
Author: Собеседник
Date: 12.12.2026
Weekday: Суббота
Time: 12:36:20
Audio 1:
  Transcription: длинный монолог про проект
```

### Фото

```

Date: 12.12.2026
Weekday: Суббота
Time: 12:37:02

  Description: рыжий корги на траве
```

```
Time: 12:38:10

  Description: девушка катается на сноуборде с горы
```

```
  Transcription: только не пугайся

  Transcription: можешь приехать на выходных?
```

## Граничные случаи и решения по ним

**У сообщения нет ни текста, ни вложений.** Печатаем `Text: (empty)`.
Полностью пропускать сообщение нельзя — оно есть в переписке, его
надо показать.

**У вложения все подполя пустые** (например, фото без описания, потому
что описать не удалось и поле `description` пустое, а `Description: (failed)`
не пишем — описание ещё не запускалось). В этом случае вложение не
печатается совсем. Если у сообщения только такое вложение и нет текста
— получится блок с одним `Text: (empty)`.

**У одного сообщения несколько `Text:`.** Не бывает. У сообщения один
текст, одно поле `Text`. Несколько `Text:` подряд означало бы несколько
сообщений — и тогда они идут отдельными блоками.

**Сообщение помечено как удалённое.** В историю не попадает. Удалённые
сообщения отфильтровываются на уровне SQL-запроса (`deleted_at IS NULL`),
до сборки истории.

**Очень старое сообщение с зависшим `pending`-статусом обработки.**
В историю попадает с `(failed)` в соответствующем подполе. См. правило 5.
