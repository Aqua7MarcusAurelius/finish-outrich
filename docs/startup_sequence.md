# Последовательность запуска системы

> Описывает в каком порядке что стартует и что должно быть готово
> перед тем как следующий компонент начинает работу.

---

## Полная последовательность

```
[Docker Compose]
    │
    ├── PostgreSQL  → healthcheck пройден
    ├── Redis       → healthcheck пройден
    └── MinIO       → healthcheck пройден
         │
         ▼
[Приложение стартует]
    │
    ├── 1. Alembic  → применяет миграции базы данных
    │                 первый запуск — создаёт все таблицы и заполняет settings дефолтами
    │                 повторный запуск — применяет только новые изменения
    │
    ├── 2. MinIO    → создаёт bucket если его нет
    │
    ├── 3. FastAPI  → API принимает запросы на localhost:8000
    │
    ├── 4. Модули   → подписываются на шину событий, готовы обрабатывать события
    │                 history, transcription, media_description
    │
    └── 5. Менеджер воркеров → готов, ждёт команд через API

[Система готова]
    └── воркеры запускаются вручную через дашборд или API
```

---

## Детали каждого шага

### Шаг 1 — Alembic (миграции)
Запускается автоматически при старте приложения до того как FastAPI начинает принимать запросы.

```bash
alembic upgrade head
```

- Первый запуск → создаёт все таблицы включая `settings` с дефолтными значениями и `events_archive`
- Повторный запуск → проверяет текущую версию схемы, применяет только новые миграции
- Если миграция упала → приложение не стартует, ошибка в stdout контейнера

Папка с миграциями: `alembic/versions/`

### Шаг 2 — MinIO bucket
Создаём bucket `tgframework` если его нет. Название bucket берётся из `.env` (`MINIO_BUCKET`).

### Шаг 3 — FastAPI
API начинает принимать запросы. До этого момента любой запрос вернёт ошибку.

### Шаг 4 — Модули
Каждый модуль подписывается на нужные ему события шины. Порядок подписки не важен — шина хранит события и модуль прочитает всё что пропустил.

### Шаг 5 — Менеджер воркеров
Читает таблицу `accounts` из базы. Видит все авторизованные аккаунты. Ждёт команд через API — воркеры не запускает автоматически.

## Как работает Alembic

**Создать новую миграцию после изменения таблицы:**
```bash
alembic revision --autogenerate -m "добавил поле X в таблицу Y"
```

**Применить все миграции:**
```bash
alembic upgrade head
```

**Откатить последнюю миграцию:**
```bash
alembic downgrade -1
```

**Посмотреть историю миграций:**
```bash
alembic history
```

## Структура папки alembic

```
alembic/
├── versions/                ← файлы миграций (создаются командой alembic revision)
│   ├── 0001_initial.py      ← первая миграция — создаёт все таблицы и заполняет settings
│   └── 0002_*.py            ← следующие миграции по мере развития проекта
└── env.py                   ← конфигурация Alembic, подключение к БД
alembic.ini                  ← настройки Alembic
```

## Если что-то пошло не так при старте

Поскольку шина событий ещё не готова на этапах 1-3, инфраструктурные ошибки старта пишутся в stdout контейнера. Это единственный случай когда лог идёт не через шину.

- **Postgres не поднялся** — Docker Compose держит `depends_on: condition: service_healthy`, приложение не стартует до готовности БД
- **Alembic упал** — ошибка в stdout, контейнер падает, `restart: unless-stopped` попытается поднять снова
- **MinIO bucket не создался** — ошибка в stdout, контейнер падает
- **FastAPI не поднялся** — ошибка в stdout

Все последующие ошибки (после готовности API) идут через шину.

---

## Принципы которые мы зафиксировали

1. Приложение стартует только после того как PostgreSQL, Redis и MinIO готовы
2. Alembic запускается автоматически при старте — до FastAPI
3. Если миграция упала — приложение не стартует
4. Воркеры при старте системы не запускаются автоматически — только вручную
5. Модули подписываются на шину при старте — порядок подписки не важен
6. Инфраструктурные ошибки на этапе старта — в stdout (шина ещё не готова)
