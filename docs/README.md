# Telegram Automation Framework

> Базовый шаблон для проектов автоматизации Telegram-клиентов.
> Пул Telegram-аккаунтов под одним управлением, с записью истории диалогов,
> транскрибацией голосовых, описанием медиа и единым веб-интерфейсом.

---

## О проекте

Система управляет несколькими Telegram-аккаунтами (воркерами), каждый из которых:
- Подключается к Telegram через собственный SOCKS5-прокси
- Пишет всю историю личных переписок в PostgreSQL
- Сохраняет медиафайлы в MinIO на 3 дня (метаданные — навсегда)
- Автоматически транскрибирует голосовые и видео-сообщения (Whisper)
- Автоматически описывает фото, стикеры, GIF, видео и документы (GPT-4o)

Компоненты общаются через единую шину событий на Redis Streams — она же является основным логом системы. Всё управление — через FastAPI + веб-дашборд.

**Область применения:** только личные диалоги. Группы и каналы не поддерживаются — это сознательное упрощение модели данных.

## Технологический стек

| Слой | Технология |
|---|---|
| Язык | Python 3.x |
| API | FastAPI |
| БД | PostgreSQL 16 (через asyncpg — без ORM) |
| Миграции | Alembic |
| Кэш / шина | Redis 7 (Redis Streams для шины) |
| Объектное хранилище | MinIO (S3-совместимое) |
| Telegram-клиент | Telethon |
| AI-провайдер | OpenRouter (Whisper, GPT-4o) |
| Обработка видео | FFmpeg |
| Инфраструктура | Docker + Docker Compose |
| Реалтайм для UI | SSE (Server-Sent Events) |

## Быстрый старт (локально)

Предполагается что установлены Docker и Docker Compose.

```bash
# 1. Клонируем репозиторий
git clone <repo-url>
cd tg-framework

# 2. Готовим .env
cp .env.example .env
# Отредактировать .env — проставить пароли PostgreSQL, MinIO, API_TOKEN, OPENROUTER_API_KEY

# 3. Поднимаем окружение
docker-compose up -d

# 4. Проверяем что API жив
curl http://localhost:8000/system/health
# Ожидаем: { "status": "ok", "components": { ... } }
```

После этого:
- API и документация доступны на `http://localhost:8000/docs`
- Веб-консоль MinIO — на `http://localhost:9001`
- Дашборд (когда будет написан) — отдельный фронт, подключается к API

Первый аккаунт авторизуется через форму в дашборде или напрямую через `POST /auth/start` → `POST /auth/code` → (при 2FA) `POST /auth/2fa`. После этого запускается воркер через `POST /workers/{account_id}/start`.

## С чего начать кодить

Порядок реализации от фундамента к модулям:

**Этап 1 — Инфраструктура**
1. `docker-compose.yml` + `Dockerfile` (см. `docker.md`)
2. `core/config.py` — загрузка `.env`
3. `core/db.py` — подключение к PostgreSQL через asyncpg
4. `core/redis.py` — подключение к Redis
5. `core/minio.py` — подключение к MinIO
6. `alembic/` — инициализация + первая миграция со всеми таблицами и дефолтами `settings`

**Этап 2 — Шина и базовый API**
7. `core/bus.py` — публикация и чтение Redis Streams + запись в `events_archive`
8. `api/main.py` — FastAPI запускается, `/system/health` и `/system/stats` работают
9. `api/sse.py` — базовая инфраструктура SSE (Last-Event-ID, heartbeat)

**Этап 3 — Воркер и Telegram**
10. `modules/worker/wrapper.py` — враппер с первым набором команд (см. `modules/wrapper.md`)
11. `modules/auth/` — модуль авторизации + endpoints `/auth/*`
12. `modules/worker_manager/` — управление воркерами + endpoints `/workers/*`
13. `modules/worker/worker.py` — жизненный цикл воркера

**Этап 4 — История**
14. `modules/history/service.py` — запись сообщений в БД
15. `modules/history/cleaner.py` — чистильщик MinIO
16. `modules/history/routes.py` — endpoints `/accounts/{id}/dialogs`, `/dialogs/*`, `/messages/*`
17. `modules/history_sync/service.py` — нагон при старте воркера

**Этап 5 — Медиа**
18. `modules/transcription/service.py` — OpenRouter → Whisper
19. `modules/media_description/service.py` — FFmpeg + OpenRouter → GPT-4o

**Этап 6 — UI-поддержка**
20. `api/routes/events.py` — архив, фильтры, экспорт, SSE `/events/stream`
21. `api/routes/search.py` — глобальный поиск
22. `api/routes/system.py` — `/dashboard`, `/system/proxy-check`

Дашборд (фронтенд) — отдельная задача, после того как API полностью работает.

## Принципы проекта

Полный список — в концах каждого документа. Ключевые:

1. **Модули не знают друг о друге — только о шине.** Любое взаимодействие через event bus.
2. **Шина только для уведомлений.** За данными — в Postgres/MinIO/Telegram напрямую.
3. **Шина — основной лог системы.** Все ошибки публикуются туда. Отдельного логирования приложения нет.
4. **В каждую таблицу пишет один модуль.** Никаких гонок.
5. **Любое общение с Telegram — только через враппер.**
6. **Один аккаунт + один собеседник = один диалог.** Аккаунты изолированы.
7. **ORM не используем — SQL руками через asyncpg.**
8. **Система работает только с личными диалогами.**
9. **Миграции через Alembic — автоматически при старте.**
10. **Падение одного воркера не влияет на остальные.**

## Что отложено

- **Симулятор человечности** — задержки чтения, паузы набора, паузы в нагоне. Закрывает темы FloodWaitError и банов
- **Управление модулями per-account** — выключать отдельные модули для конкретных аккаунтов
- **Отправка медиа** (`send_media` + UI) — добавим когда понадобится
- **Открытие диалога на конкретном сообщении** из поиска — когда понадобится

## Известные ограничения MVP

Зафиксированы в `architecture.md` в разделе «Известные ограничения». Кратко:

- Идемпотентность публикаций на шину держится только на уникальных индексах БД
- Batch retry массово сбойных медиа не реализован (только ручной через API)
- Rate limits Telegram закрываются в будущем через симулятор человечности
- Всё приложение в одном контейнере — при росте нагрузки потребуется разделение
- Ротация `API_TOKEN` ручная через `.env`
