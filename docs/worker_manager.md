# Модуль менеджера воркеров

> Управляет пулом воркеров — запускает, останавливает, удаляет.
> Работает постоянно, управляется через API.

---

## Принцип работы

Менеджер воркеров видит все авторизованные аккаунты в базе и управляет воркерами для них. Воркер существует только для аккаунта который есть в базе. Менеджер не знает про внутреннюю логику воркеров — он только запускает и останавливает их.

```
[API]
  └── команда пользователя
       └── [Менеджер воркеров]
            ├── смотрит в accounts — какие аккаунты есть
            ├── запускает / останавливает / удаляет воркеры
            └── следит за состоянием воркеров через Redis
```

---

## Команды через API

| Команда | Что делает |
|---|---|
| `worker.start` | Запускает воркер для аккаунта |
| `worker.stop` | Останавливает воркер аккуратно |
| `worker.delete` | Полностью удаляет аккаунт и всю его историю |
| `worker.list` | Показывает все аккаунты и статус их воркеров |

Подробнее по соответствующим API endpoints — в `api.md`, раздел «Воркеры».

---

## Детали каждой команды

### worker.start
- Проверяет что аккаунт есть в базе и `is_active = true`
- Проверяет что воркер для этого аккаунта ещё не запущен
- Запускает воркер — без перезапуска всей системы
- Записывает статус `running` в Redis

### worker.stop
- Посылает воркеру сигнал остановки
- Воркер аккуратно завершает текущие задачи и останавливается
- Статус в Redis → `stopped`

### worker.delete
Полное удаление — необратимо. Порядок важен:

```
1. Останавливаем воркер если запущен
2. Удаляем все медиафайлы аккаунта из MinIO
3. Удаляем из БД каскадно: accounts → dialogs → messages → media / reactions / message_edits
4. Удаляем статус из Redis
5. Публикуем на шину account.deleted
```

### worker.list
Возвращает список всех аккаунтов из базы с их текущим статусом из Redis:

```
[
  { id: 1, name: "Аккаунт для прогрева", phone: "+7999...", status: "running" },
  { id: 2, name: "Тестовый",             phone: "+7888...", status: "stopped" },
  { id: 3, name: "Резервный",            phone: "+7777...", status: "crashed" }
]
```

---

## Если воркер упал сам

Воркер может упасть по любой причине — ошибка в коде, проблема с сетью и тд. Менеджер это замечает через Redis — статус воркера перестал обновляться или стал `crashed`.

```
Воркер упал
    └── менеджер видит статус crashed в Redis
    └── публикует system.error на шину
    └── делает одну попытку перезапуска
         ├── успешно → статус running, продолжает работу
         └── снова упал → статус crashed, больше не трогает
                       → публикует system.error на шину
                       → пользователь видит через worker.list и в логе
                       → решает сам что делать
```

Автоматически больше одного раза не перезапускает — чтобы не создавать петлю если проблема системная.

---

## Обработка ошибок

Все ошибки менеджера публикуются на шину событий со статусом `error` или как `system.error` для инфраструктурных проблем. Отдельного логирования не ведётся.

---

## Принципы которые мы зафиксировали

1. Менеджер управляет только аккаунтами которые есть в базе
2. Удаление — полное и необратимое, включая всю историю диалогов и файлы в MinIO
3. Остановка — аккуратная, воркер завершает текущие задачи
4. При падении воркера — одна попытка перезапуска, потом не трогаем
5. Статусы воркеров живут в Redis — оперативные данные, не в базе
6. Все падения и ошибки перезапуска публикуются на шину как `system.error`
