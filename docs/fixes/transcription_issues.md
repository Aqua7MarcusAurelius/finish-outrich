# Transcription Service — анализ проблем

**Дата:** 2026-04-22  
**Исходные файлы:** `modules/transcription/service.py`, `core/openrouter.py`, `modules/transcription/ffmpeg.py`

---

## Критические проблемы

### 1. Очередь зависает при ошибках OpenRouter

**Файл:** `modules/transcription/service.py`

Если OpenRouter вернул ошибку (5xx, таймаут, сеть упала) — событие не ACK'ается. Redis Streams оставляет его в pending и переподаёт снова. Повторная попытка происходит немедленно, без задержки. При нестабильном OpenRouter это превращается в busy-loop на одном событии, который блокирует обработку следующих 19 событий в батче.

**Решение:** добавить экспоненциальный backoff перед retry и максимальное число попыток на одно событие, после которого событие принудительно ACK'ается с `status="failed"`.

---

### 2. Нет backoff для HTTP 429 (rate limit)

**Файл:** `core/openrouter.py`

На `OpenRouterError` retry происходит немедленно. Если OpenRouter вернул `429 Too Many Requests` — повторный запрос улетает сразу и гарантированно получает ещё один `429`. Не учитывается заголовок `Retry-After`.

**Решение:** парсить код ошибки; при `429` и `503` ждать значение из `Retry-After` (или fallback: 5–30–60 сек экспоненциально).

---

### 3. Неограниченное количество retries

**Файл:** `modules/transcription/service.py`, `_call_with_retries()`

`retries` читается из БД при каждом задании без какого-либо ограничения в коде. Если admin выставит `settings.transcription.retries = 100` — сервис сделает 101 попытку подряд на каждом файле.

**Решение:** захардкодить cap: `attempts = max(1, min(1 + retries, MAX_RETRIES))` где `MAX_RETRIES = 5`.

---

### 4. Двойной retry при пустом результате маскирует ошибки

**Файл:** `modules/transcription/service.py`, ~строки 310–317

Если Whisper вернул пустую строку, код делает скрытый второй вызов вне основного retry-цикла. Если этот вызов упал с ошибкой — возвращается `status="done"` с пустым текстом. Итого: тишина и сетевая ошибка имеют одинаковый исход, различить их невозможно.

**Решение:** убрать скрытый retry или ввести отдельный статус (`status="done_empty"` / флаг `"is_silence": true`).

---

### 5. Настройки читаются из БД на каждое задание

**Файл:** `modules/transcription/service.py`, `_get_retries()`

Каждый job делает отдельный `SELECT` за значением `retries`. При нагрузке — лишние запросы. При зависании БД — все задания в батче блокируются.

**Решение:** кешировать значение с TTL 60 сек.

## Главный вывод

Проблемы #1 + #2 вместе: при нестабильном OpenRouter или rate limit сервис уходит в бесконечный tight-loop на зависшем событии без backoff. Пока событие не обработается, очередь де-факто стоит. Проявится под нагрузкой или при первом перебое у OpenRouter.
