# Модуль авторизации

> Выполняется один раз для каждого нового аккаунта.
> Интерактивный процесс — требует ввода от пользователя на нескольких шагах.
> Управляется через API.

---

## Принцип работы

Модуль авторизации — это отдельный процесс который не связан с воркерами. Он запускается по команде через API, проводит пользователя через все шаги авторизации и в конце записывает готовый аккаунт в базу. После этого менеджер воркеров видит новый аккаунт и может запустить для него воркер.

```
Пользователь → API → Модуль авторизации → Telegram
                                        ← код на телефон
Пользователь → API → Модуль авторизации → завершает авторизацию
                                        → сохраняет в БД
                                        → менеджер воркеров видит новый аккаунт
```

Детали API endpoints — в `api.md`, раздел «Авторизация аккаунтов».

## Шаги авторизации

### Шаг 1 — Получаем данные
Через API принимаем:
- Номер телефона
- Основной прокси (`socks5://user:pass@host:port`)
- Запасной прокси (`socks5://user:pass@host:port`)
- Имя аккаунта (для удобного различения)

### Шаг 2 — Проверяем оба прокси
До любых действий — проверяем что оба прокси работают. Если хотя бы основной недоступен — останавливаемся и возвращаем ошибку. Нет смысла начинать авторизацию через нерабочий прокси.

```
Проверка прокси 1 → не работает → стоп, ошибка пользователю
Проверка прокси 2 → не работает → стоп, ошибка пользователю
Оба работают → продолжаем
```

### Шаг 3 — Подключаемся и запрашиваем код
Подключаемся к Telegram через основной прокси и запрашиваем код подтверждения. Весь дальнейший процесс идёт через этот же прокси — Telegram должен видеть один и тот же IP на всех шагах.

Возвращаем пользователю через API: "код отправлен, введите его".

### Шаг 4 — Принимаем код
Пользователь вводит код через API. Если код неверный — даём ещё одну попытку. Telegram аннулирует коды примерно через 10 минут — если время вышло, начинаем сначала с шага 3.

### Шаг 5 — 2FA (если включена)
Если на аккаунте включена двухфакторная аутентификация — запрашиваем пароль через API и передаём его в Telegram.

### Шаг 6 — Сохраняем результат
Авторизация завершена. Сохраняем в таблицу `accounts`:
- Имя аккаунта
- Номер телефона
- Данные сессии Telethon (`session_data` в байтах)
- Основной и запасной прокси
- `is_active = true`

Публикуем на шину событие `account.created`.

## Состояние между шагами

Сессия авторизации живёт в Redis по ключу `auth_session:{session_id}` с TTL 15 минут (Telegram аннулирует код примерно через 10 минут, плюс запас на 2FA).

Содержимое:
- `phase` — текущая фаза (`code_sent` / `2fa_required` / `done` / `failed`)
- `phone`, `name`, `proxy_primary`, `proxy_fallback`
- `created_at`, `error`

Сам объект `TelegramClient` с открытым подключением живёт в памяти процесса модуля авторизации, индексируется по `session_id`. Если процесс перезапустился — активные авторизации теряются, клиент начинает сначала.

## Переавторизация протухшей сессии

### Как воркер определяет что сессия протухла

Воркер при старте или во время работы получает от Telegram ошибку авторизации. В этом случае:

```
Воркер → получил ошибку сессии от Telegram
       → публикует на шину account.session_expired
       → публикует system.error
       → останавливается
       → устанавливает is_active = false в accounts
```

Пользователь видит через API что аккаунт требует переавторизации.

### Причины протухания сессии

- Telegram сам завершил сессию (долгое отсутствие активности или подозрительное поведение)
- Прокси сменил IP — Telegram мог сбросить сессию
- Аккаунт заблокирован — переавторизация не поможет, воркер сообщает об этом отдельно

### Как переавторизовать

Тот же флоу что и при первой авторизации — через тот же модуль и тот же API. Отличие одно: если аккаунт с таким номером уже есть в базе — новая сессия перезаписывает старую, остальные данные (прокси, имя) остаются.

```
Пользователь → API: "переавторизовать аккаунт с номером +7999..."
             → модуль авторизации запускает флоу заново
             → после успеха перезаписывает session_data в accounts
             → устанавливает is_active = true
             → публикует account.reauthorized на шину
             → менеджер воркеров может снова запустить воркер
```

## Обработка ошибок

Все ошибки публикуются на шину:
- Ошибки флоу авторизации (неверный код, неверный 2FA, прокси упал) — события со статусом `error`
- Инфраструктурные ошибки — `system.error`

---

## Принципы которые мы зафиксировали

1. Авторизация всегда через прокси — оба прокси проверяются до начала
2. Весь флоу авторизации идёт через один и тот же прокси — Telegram видит один IP
3. Неверный код — даём ещё одну попытку
4. Истёкший код — начинаем с запроса кода заново
5. Сессия хранится в `accounts.session_data` в байтах
6. Воркер сам определяет протухание сессии и сигнализирует через шину
7. Переавторизация — тот же флоу, новая сессия перезаписывает старую
8. Состояние сессии авторизации живёт в Redis с TTL 15 минут
