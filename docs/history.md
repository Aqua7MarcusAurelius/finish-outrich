# Модуль записи истории

> Слушает шину событий и записывает все новые сообщения в базу данных.
> Работает постоянно пока воркер активен.

> Структура таблиц БД описана отдельно — см. `database_schema.md`.

---

## Главный принцип модуля

**Один наш аккаунт + один собеседник = один диалог.** Всё просто и по-человечески: я общаюсь с Олегом — это один диалог. Если тот же Олег пишет на второй наш аккаунт — это **никак не связанный** с первым диалог, даже tg_id собеседника там тот же. Аккаунты живут изолированно.

Это ключевое упрощение всей схемы. Благодаря ему:
- Не нужна отдельная таблица людей — все данные о собеседнике живут в строке `dialogs`
- В сообщениях не нужен `sender_id` — участников всего двое, хватает флага "наше/не наше"
- Безопасность — данные одного аккаунта не текут в таблицы другого через общие записи

---

## Когда запускается

При каждом старте воркера — параллельно с модулем синхронизации истории (нагоном).

---

## Что слушает на шине

| Событие | Что делает |
|---|---|
| `message.received` | Записывает новое сообщение в базу, публикует `message.saved` |
| `transcription.done` | Добавляет текст транскрипции к соответствующему медиа |
| `description.done` | Добавляет описание к соответствующему медиа |

---

## Что публикует на шину

| Событие | Когда |
|---|---|
| `message.saved` | После записи сообщения и всех его медиа в базу |
| `message.updated` | После обновления записи (транскрипция, описание, редактирование) |
| `file.cleaned` | После удаления файла из MinIO чистильщиком |

---

## Событие `message.saved` — ключевое для медиа-модулей

К моменту публикации этого события запись в БД гарантированно существует. Медиа-модули (транскрибация, описание) слушают именно его, а не `message.received` — чтобы не было гонки.

Payload:

```
type: message.saved
parent_id: 001           ← ссылка на message.received
data: {
  message_id: 42              ← наш внутренний id
  telegram_message_id: 10005
  dialog_id: 7
  is_outgoing: false

  has_text: false
  has_audio: true             ← voice / video_note / audio
  has_image: false            ← photo / sticker / gif
  has_video: false            ← видеофайл (требует и транскрипции, и описания)
  has_document: false         ← pdf, xlsx и т.п.

  media: [
    {
      media_id: 15
      type: voice
      storage_key: "account_1/dialog_7/10005.ogg"
      duration: 12
      mime_type: "audio/ogg"
    }
  ]
}
```

## Поток записи нового сообщения

```
[Шина: message.received]
    │
    ▼
[Модуль истории]
    ├── проверяет дубли по (dialog_id, telegram_message_id)
    ├── находит или создаёт запись в dialogs
    ├── пишет запись в messages
    ├── для каждого вложения — пишет запись в media
    ├── для каждой реакции — пишет запись в reactions
    │
    └── публикует message.saved (parent_id → message.received)
         │
         ├──▶ модуль транскрибации (если has_audio/has_video)
         └──▶ модуль описания медиа (если has_image/has_video/has_document)
```

---

## Поток обновления медиа

```
[Шина: transcription.done или description.done]
    │
    ▼
[Модуль истории]
    ├── находит запись в media по media_id
    ├── пишет результат в transcription или description
    ├── обновляет статус (done / failed)
    │
    └── публикует message.updated
```

## Чистильщик файлов

Часть модуля истории. Политика: файлы живут 3 дня, метаданные — навсегда.

- Частота: раз в час (`settings.cleaner.interval_hours`)
- Пачка: до 50 записей за прогон (`settings.cleaner.batch_size`)
- Возраст файла: 3 дня (`settings.cleaner.file_ttl_days`)
- Не трогает файлы со статусом `pending` у транскрибации или описания
- При ошибке MinIO — не обновляет БД, пробует в следующий прогон
- Публикует `file.cleaned` на шину для аудита
- При инфраструктурных ошибках — `system.error`

После удаления файла в `media`:
- `storage_key` → null
- `file_deleted_at` → текущее время
- `transcription` и `description` остаются — текст важнее чем исходный файл

## Обработка сложных случаев

**Редактирование сообщения.**
- В `messages.text` пишем новый текст
- Старую версию сохраняем в `message_edits`
- Проставляем `edited_at`
- Публикуем `message.updated`

**Удаление сообщения собеседником.**
Soft delete — запись не стираем. Ставим `deleted_at`. Мы видели сообщение, контекст разговора на него опирается — физически не убираем.

**Системный чат Telegram (777000).**
Игнорируем целиком. Не создаём диалог, не пишем сообщения.

**Защита от дублей.**
Уникальный индекс `(dialog_id, telegram_message_id)` на `messages` гарантирует это на уровне БД. Модуль перед вставкой всё равно проверяет — чтобы не ловить исключение на каждом дубле от нагона.

---

## Обработка ошибок

Все ошибки публикуются на шину:
- Ошибка записи конкретного сообщения/медиа — событие со статусом `error` и `parent_id`
- Инфраструктурные (БД недоступна, Redis недоступен) — `system.error`
- Чистильщик не смог удалить файл — `system.error` и повтор в следующем прогоне

## Принципы которые мы зафиксировали

1. Один аккаунт + один собеседник = один диалог. Аккаунты изолированы.
2. Все данные о собеседнике в `dialogs` — таблицы людей нет.
3. В личном чате достаточно `is_outgoing` вместо `sender_id`.
4. Данные пересыльщиков — снимок на момент получения (денормализация).
5. Вложения в отдельной таблице `media`.
6. Файлы в MinIO по ключу `{account_id}/{dialog_id}/{telegram_message_id}.{ext}`.
7. `transcription` и `description` — независимые поля, храним как есть без обрезки.
8. Файлы живут 3 дня, метаданные навсегда.
9. Чистильщик: раз в час, до 50 записей, не трогает pending.
10. Редактирование: новый текст в `messages`, старые версии в `message_edits`.
11. Удаление: soft delete через `deleted_at`.
12. Реакции с историей. Два поля: `emoji` и `custom_emoji_id`.
13. Защита от дублей по `(dialog_id, telegram_message_id)`.
14. Медиа-модули слушают `message.saved`, не `message.received`.
15. Системный чат Telegram (777000) игнорируем целиком.
16. Сессия Telethon хранится в `accounts.session_data` в байтах — надёжнее чем на диске.
17. Прокси хранятся строкой `socks5://user:pass@host:port`.
18. Статус воркера — только в Redis, не в базе.
19. Все ошибки модуля публикуются на шину, отдельного логирования нет.
