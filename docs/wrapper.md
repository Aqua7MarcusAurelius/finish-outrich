# Враппер Telegram (Telegram Wrapper)

> Единственная точка общения с Telegram внутри воркера.
> Никто не обращается к Telegram напрямую — только через враппер.

---

## Принцип работы

Враппер — это API воркера. У него есть чёткий список команд. Любой модуль который хочет что-то сделать с Telegram — вызывает команду враппера и получает результат.

**Почему это важно:**
Если что-то изменится в Telethon или в логике работы с Telegram — меняется только враппер. Все остальные модули этого не замечают.

```
[Любой модуль]
    └── вызывает команду враппера
         └── [Враппер]
              └── общается с Telegram через Telethon
              └── возвращает результат модулю
              └── публикует событие на шину
```

---

## Команды враппера

| Команда | Что делает |
|---|---|
| `send_message` | Отправить текстовое сообщение |
| `send_media` | Отправить медиафайл |
| `read_message` | Отметить сообщение как прочитанное |
| `get_dialogs` | Получить список всех диалогов |
| `get_history` | Получить историю диалога кусками |
| `resolve_username` | Резолв `@username` → entity (для автодиалогов) |
| `set_typing` / `cancel_typing` | Включить / снять индикатор «печатает» у собеседника |

Плюс handler `events.ChatAction` с `typing=True` — публикует событие `dialog.typing_observed` на шину. Нужно модулю AutoChat для отслеживания тайпинга собеседника и перезапуска reply-таймера.

> Список команд будет расширяться по мере добавления новых модулей.

---

## Управление прокси

Каждый воркер работает через свой прокси. Враппер следит за его состоянием.

```
Основной прокси → недоступен → Запасной прокси → недоступен → воркер останавливается
                                                              └── публикует system.error на шину
                                                              └── is_active = false в accounts
```

---

## Определение протухшей сессии

Если Telegram возвращает ошибку авторизации (`AuthKeyError`, `SessionExpiredError`, `UserDeactivatedError` и подобные):

- Публикует `account.session_expired` на шину
- Публикует `system.error` с описанием причины
- Воркер останавливается, `is_active = false`
- Требуется переавторизация через модуль авторизации

---

## Обработка ошибок

Все ошибки враппера публикуются на шину:

- **Ошибки в рамках события** (не удалось отправить сообщение, файл не скачался) — событие со статусом `error` и `parent_id` на родительское событие
- **Системные ошибки** (прокси недоступны, сессия протухла) — событие типа `system.error`

Отдельного логирования враппер не делает — шина покрывает всё.

---

## Принципы которые мы зафиксировали

1. Любое общение с Telegram — только через враппер
2. У враппера чёткий список команд
3. Изменения в логике Telegram затрагивают только враппер, не другие модули
4. Все ошибки враппер публикует на шину — `error` для событий, `system.error` для инфраструктуры
