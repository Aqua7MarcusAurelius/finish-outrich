# Модуль описания медиа

> Генерирует текстовое описание визуального контента из Telegram.
> Результат пишется в поле `media.description` через модуль истории.

---

## Вход

Модуль слушает событие `message.saved` на шине. Фильтрует по флагам:

```
if has_image or has_video or has_document:
    → обрабатывать
```

**Важно:** кружок (`video_note`) попадает и в транскрибацию (аудиодорожка) и сюда (визуальные кадры) — это два независимых процесса, два независимых поля в `media`.

Из payload события модуль получает:
- `media_id` — идентификатор записи в таблице `media`
- `storage_key` — ключ файла в MinIO
- `type` — тип медиа
- `mime_type`, `file_size` — метаданные

Модуль слушает именно `message.saved` — к этому моменту запись в `media` гарантированно существует.

---

## Выход

Модуль публикует на шину событие `description.done` с `media_id` и текстом описания. **В БД сам не пишет** — обновлением `media.description` занимается модуль истории.

Принцип: в каждую таблицу пишет только один модуль — тот что ей владеет.

---

## Ядро — что и чем обрабатываем

| Тип | Подход | Инструмент |
|---|---|---|
| Фото | байты → GPT-4o | OpenRouter → GPT-4o |
| Стикер | байты → GPT-4o | OpenRouter → GPT-4o |
| GIF | FFmpeg → 3-5 кадров → GPT-4o | FFmpeg + OpenRouter → GPT-4o |
| Видео файл | FFmpeg → 3-5 кадров → GPT-4o | FFmpeg + OpenRouter → GPT-4o |
| Кружок (video_note) | FFmpeg → 3-5 кадров → GPT-4o | FFmpeg + OpenRouter → GPT-4o |
| Документ (PDF, xlsx и др.) | байты → GPT-4o | OpenRouter → GPT-4o |

### Логика нарезки кадров через FFmpeg
Для GIF, видео и кружков нарезаем равномерно по длине — 3-5 кадров на весь файл. Количество кадров настраивается в `settings.description.frames_count`. Кадры передаются в GPT-4o как набор изображений в одном запросе. Временных файлов не создаём — кадры живут в памяти.

### Промпт для документов
Для PDF, xlsx и других файлов используем промпт:
> "Опиши что это за файл, что в нём содержится, передай общую суть."

---

## Как получаем файл

Скачиваем из MinIO через `storage.get(storage_key)` — байтами. Временных локальных файлов не создаём. Для FFmpeg передаём байты напрямую через stdin.

---

## Правила обработки

**Размер файла** — не ограничиваем.

**Очередь параллельности** — не используем, подаём запросы как получаем события.

**Ретраи при ошибке OpenRouter** — 1 повторная попытка. Если снова ошибка — `description.done` со статусом `failed`. Количество попыток настраивается в `settings.description.retries`.

**Пустой результат** — 1 повторная попытка. Если снова пусто — записываем пустой результат со статусом `done`.

---

## Поток работы

```
[Шина: message.saved с has_image / has_video / has_document = true]
    │
    ▼
[Модуль описания медиа]
    ├── читает media_id, storage_key, type из события
    ├── storage.get(storage_key) → байты файла
    ├── публикует description.started (parent_id → message.saved)
    │
    ├── фото / стикер / документ → байты → GPT-4o
    ├── gif / видео / кружок → FFmpeg → 3-5 кадров в памяти → GPT-4o
    │
    ├── ошибка → 1 повторная попытка → снова ошибка → description.done (failed)
    ├── пустой результат → 1 повторная попытка → снова пусто → description.done (done, текст пустой)
    │
    └── публикует description.done (parent_id → description.started, media_id + текст)
         │
         ▼
    [Модуль истории]
    └── находит запись по media_id, пишет текст в media.description, ставит description_status = done
```

---

## Повторный запуск описания

Через API endpoint `POST /media/{id}/redescribe`:
- Сбрасывает `description_status` в `pending`
- Публикует `media.reprocess.requested` на шину
- Модуль подхватывает и проходит флоу заново

---

## Обработка ошибок

Все ошибки публикуются на шину:
- Ошибка OpenRouter, FFmpeg упал, файл не найден — событие `description.done` со статусом `error`
- Инфраструктурные (MinIO целиком недоступен) — `system.error`

---

## Принципы которые мы зафиксировали

1. Модуль слушает `message.saved` — запись в `media` гарантированно есть
2. Фильтрация по флагам `has_image` / `has_video` / `has_document` — без похода в БД
3. Файл забираем из MinIO байтами, временных файлов не создаём
4. GIF / видео / кружок — нарезаем 3-5 кадров через FFmpeg в памяти
5. Документы — передаём в GPT-4o с промптом на общее описание содержимого
6. Кружок обрабатывается и здесь и в транскрибации — два независимых поля
7. В БД не пишем сам — результат на шину, история дописывает в `media`
8. Ретраи: 1 попытка при ошибке, 1 повторная попытка при пустом результате
9. Количество ретраев и кадров настраивается в `settings` без перезапуска
