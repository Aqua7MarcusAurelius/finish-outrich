# Архитектура проекта: Telegram Automation Framework

> Базовый шаблон для проектов автоматизации Telegram-клиентов.
> Документ фиксирует только обсуждённые и принятые решения.

---

## Общая схема

```
[Дашборд — веб-интерфейс]
  └── основной клиент API
  └── два раздела: технический монитор и просмотр чатов
       │
       │ HTTP REST + SSE
       ▼
[API — FastAPI]
  └── единая точка управления всей системой
  └── не общается с модулями напрямую — читает Redis, команды передаёт менеджеру
  └── документация на localhost:8000/docs
       │
       ├──▶ [Модуль авторизации]
       │     └── проверяет прокси → авторизует через Telegram
       │     └── сохраняет сессию и аккаунт в PostgreSQL
       │
       ▼
[Менеджер воркеров]
  └── запускает / останавливает / удаляет воркеры без перезапуска системы
  └── видит все доступные аккаунты в базе
  └── при падении воркера — одна попытка перезапуска
       │
       ▼
[Воркер аккаунта 1..N]
  └── [Враппер Telegram] ← единственная точка общения с Telegram
  └── управление прокси (основной → запасной → стоп)
       │
       ├── при старте
       │    ├── [Модуль нагона]         → синхронизирует диалоги и историю
       │    └── [Модуль записи истории] → слушает шину, пишет новые сообщения
       │
       ├── медиафайлы при получении → [MinIO]
       │
       └── публикует события на шину
            │
            ▼
[Шина событий — Redis Streams + архив в PostgreSQL]
└── только уведомления — не для запросов данных
└── все события видны как лог — шина является основным логом системы
└── все ошибки публикуются сюда — отдельного логирования нет
└── API читает шину и транслирует в SSE потоки для дашборда
     │
     ├── [Модуль записи истории] ← слушает message.received
     │    └── публикует message.saved с флагами типов медиа
     │         │
     │         ├──▶ [Транскрибация]   → [OpenRouter → Whisper]
     │         └──▶ [Описание медиа]  → FFmpeg (для видео/gif/кружков) → [OpenRouter → GPT-4o]
     │
     └── [Симулятор человечности] ⏸ отложен

[PostgreSQL] — постоянное хранилище: аккаунты, сессии, история диалогов, настройки, архив событий
[Redis]      — живая шина событий + статусы воркеров + состояние сессий авторизации
[MinIO]      — хранилище медиафайлов
```

---

## Область применения

**Система работает только с личными диалогами.** Группы и каналы не поддерживаются.

Это архитектурное решение на котором основаны ключевые упрощения — структура БД модуля истории (нет таблицы `users`, в сообщениях `is_outgoing` вместо `sender_id`). Если появится потребность в группах/каналах — потребуется заметная миграция.

## Компоненты

### Дашборд
Веб-интерфейс — основной клиент API. Два раздела:

**Технический монитор:**
- Карточки воркеров — статус, номер, количество диалогов, кнопки Start/Stop/Reauth
- Бегущий лог шины событий с фильтрами и экспортом
- Обновления в реальном времени через SSE

**Просмотр чатов:**
- Список аккаунтов и диалогов
- Просмотр истории сообщений с медиа, транскрипциями и описаниями
- Обновления через SSE — новые сообщения появляются без перезагрузки

При первой загрузке страницы — данные через REST. Дальше — события через SSE, база не опрашивается постоянно.

### API (FastAPI)
- Единая точка управления — интерфейс между пользователем и системой
- Не общается с модулями напрямую — читает из Redis, команды передаёт менеджеру воркеров
- Три SSE потока для дашборда: статусы воркеров, события диалога, бегущий лог
- Автоматически генерирует документацию на `localhost:8000/docs`
- SQL руками через asyncpg — ORM не используем
- Подробнее: `api.md`

### Модуль авторизации
Отдельный интерактивный процесс — выполняется один раз для каждого нового аккаунта.
- Принимает номер телефона, имя и два прокси
- Проверяет оба прокси перед началом — если хотя бы основной не работает, стоп
- Подключается к Telegram через основной прокси
- Проводит авторизацию (код + 2FA если есть) — всё через прокси
- Сохраняет сессию Telethon в PostgreSQL (`accounts.session_data` в байтах)
- При протухшей сессии — воркер сигнализирует, переавторизация через тот же флоу
- Подробнее: `modules/auth.md`

### Менеджер воркеров
- Видит все авторизованные аккаунты в базе
- Запускает воркер для аккаунта — без перезапуска системы
- Останавливает воркер аккуратно — ждёт завершения текущих задач
- Удаляет воркер полностью — аккаунт, история, файлы в MinIO
- При падении воркера — одна попытка перезапуска, потом не трогает
- Подробнее: `modules/worker_manager.md`

### Воркер
Изолированный процесс для одного Telegram-аккаунта. Если один воркер падает — остальные продолжают работать.

Состав воркера:
- **Враппер Telegram** — единственная точка общения с Telegram через Telethon. Подробнее: `modules/wrapper.md`
- **Управление прокси** — основной прокси, при недоступности переключается на запасной. Если оба недоступны — воркер останавливается и пишет об этом на шину

### Шина событий (Redis Streams)
Центральное место через которое все компоненты обмениваются информацией. Является основным логом системы. Подробнее: `event_bus.md`

Принципы:
- Никто не общается с другими модулями напрямую — только через шину
- Шина только для уведомлений — не для запросов данных
- Каждое событие хранится в порядке поступления
- Модуль который был недоступен — прочитает пропущенные события когда вернётся
- Все события видны как лог — для отладки и аудита
- Все ошибки публикуются сюда со статусом `error` или типом `system.error`
- Отдельного логирования приложения нет — шина покрывает всё
- API читает шину и транслирует в три SSE потока для дашборда

## Внешние сервисы

### OpenRouter
Прослойка между системой и AI сервисами. Если один сервис лёг или ключ отвалился — переключаемся на другую модель без изменения кода.

| Задача | Модель |
|---|---|
| Аудио / голосовые / кружки → текст | Whisper |
| Фото / видео / стикеры / GIF / документы → описание | GPT-4o |

## Инфраструктура

### PostgreSQL
Постоянное хранилище всего структурированного — см. `database_schema.md`.

### Redis
Быстрое хранилище:
- Шина событий (Redis Streams)
- Текущие статусы воркеров
- Состояние сессий авторизации (TTL 15 минут)

### MinIO
S3-совместимое объектное хранилище для медиафайлов:
- Фото, видео, голосовые, кружки, стикеры, GIF, документы
- Ключи по схеме `{account_id}/{dialog_id}/{telegram_message_id}.{ext}`
- Файлы хранятся 3 дня, удаляются чистильщиком модуля истории
- Метаданные остаются в PostgreSQL навсегда

### FFmpeg
Используется модулем описания медиа для нарезки кадров из видео, GIF и кружков.
- Работает в памяти, без временных файлов на диске
- Нарезает 3-5 равномерных кадров на всю длину файла

### asyncpg
Библиотека для работы с PostgreSQL. SQL пишем руками — ORM не используем.

### Alembic
Миграции базы данных. Применяются автоматически при старте приложения. Подробнее: `startup_sequence.md`.

### Docker
Все компоненты запускаются в одном контейнере. Docker Compose поднимает приложение + PostgreSQL + Redis + MinIO. Подробнее: `docker.md`.

### SOCKS5 прокси
Каждый аккаунт имеет основной и запасной прокси в формате `socks5://user:pass@host:port`. Проверяются при авторизации и при старте воркера.

### Telethon
Библиотека для работы с Telegram как клиент (не бот).

### GitHub
Публичный репозиторий для хранения кода и версий.

**Что не хранить — `.gitignore`:**
- `.env` — пароли, токены, API ключи
- Локальные конфиги с данными прокси

## Логирование

Шина событий является основным логом системы. Все значимые события и ошибки публикуются туда:

- **Бизнес-события** — `message.received`, `message.saved`, `transcription.done`, ...
- **Ошибки в рамках событий** — статус `error` + `parent_id` на родителя
- **Системные ошибки** — тип `system.error` (прокси упал, сессия протухла, OpenRouter недоступен)

Единственное исключение — **stdout контейнера**. Туда идут инфраструктурные ошибки старта и случаи когда шина сама недоступна (Redis упал целиком). Это минимальный safety net, без структурированного логирования.

## Симулятор человечности (отложен)

Отдельный будущий модуль — имитирует поведение живого пользователя: задержки прочтения, паузы набора текста, паузы между запросами у нагона. Защищает от FloodWaitError и банов аккаунтов.

Все темы про «паузы между запросами», «когда читать», «когда печатать» — стекаются в этот модуль. Пока работаем без него на минимальной разумной задержке.

## Известные ограничения (на будущее)

На этапе MVP сознательно не делаем, держим в уме:

- **Идемпотентность публикаций на шину** — если модуль дважды опубликует одно событие, защита только на уровне уникальных индексов БД. Для медиа-обработки это может приводить к лишним вызовам OpenRouter. Разберёмся если станет проблемой.
- **Batch retry сбойных `failed` медиа** — сейчас только ручная перезапись через `POST /media/{id}/retranscribe`. Массовый сценарий (OpenRouter упал на час, 500 файлов в failed) — через UI или отдельную админ-команду, по мере надобности.
- **Rate limits Telegram** — закрываются симулятором человечности. До него — минимальные разумные задержки в нагоне.
- **Разделение на несколько контейнеров** — при росте нагрузки воркеры могут выехать в отдельный контейнер. Сейчас всё в одном.
- **Ротация API_TOKEN** — токен в `.env`, меняется перезапуском. Для production позже нужен нормальный механизм.

## Принципы которые мы зафиксировали

1. Падение одного воркера не влияет на остальные
2. Система управляется через API — без перезапуска
3. Модули не знают друг о друге — только о шине
4. Шина только для уведомлений — не для запросов данных
5. Любое общение с Telegram — только через враппер
6. Все события сохраняются и видны как лог — шина является основным логом
7. Все ошибки публикуются на шину — отдельного логирования нет (кроме stdout для инфры)
8. Новый модуль добавляется без изменения существующих
9. Медиа-модули слушают `message.saved` — запись в БД и файл в MinIO гарантированно есть
10. Система работает только с личными диалогами — групп и каналов нет
11. Авторизация всегда через прокси — оба прокси проверяются до начала
12. Сессии Telethon хранятся в PostgreSQL — надёжнее чем на диске
13. При падении воркера — одна попытка перезапуска, потом не трогаем
14. Удаление аккаунта — полное и необратимое, включая историю и файлы в MinIO
15. API не общается с модулями напрямую — читает Redis, команды передаёт менеджеру воркеров
16. Дашборд — основной клиент API. При загрузке — REST, дальше — SSE
17. ORM не используем — SQL руками через asyncpg
18. В каждую таблицу пишет только один модуль — владелец
