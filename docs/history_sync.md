# Модуль синхронизации истории (Нагон)

> Запускается при старте воркера. Синхронизирует диалоги и историю сообщений
> с тем что уже есть в нашей базе данных.

---

## Когда запускается

При каждом старте воркера — параллельно с модулем записи истории.

```
Старт воркера
    ├── Модуль записи истории  → слушает шину, пишет всё новое что приходит
    └── Модуль нагона          → догружает всё что пропустили пока воркер не работал
```

Параллельная работа защищает от разрыва — пока нагон догружает старое, запись уже ловит новое.

---

## Два этапа работы

### Этап 1 — Синхронизация списка диалогов
Порядок важен — сначала синхронизируем список диалогов, потом уже историю.

```
1. Берём список диалогов из нашей базы
2. Запрашиваем список диалогов из Telegram через враппер (get_dialogs)
3. Сравниваем
4. Новые диалоги → добавляем в базу → ставим в очередь на нагон истории
```

### Этап 2 — Нагон истории по каждому диалогу

```
1. Берём диалог из очереди
2. Смотрим в базе: какой последний ID сообщения по этому диалогу?
3. Запрашиваем у Telegram через враппер (get_history) всё после этого ID
4. Telegram отдаёт кусками — берём максимум что он даёт (~100 сообщений за запрос)
5. Записываем в базу
6. Повторяем пока не догнали текущий момент
7. Публикуем на шину событие о завершении нагона по диалогу
```

---

## Важные правила

**Защита от дублей.**
Перед записью любого сообщения — проверяем есть ли оно уже в базе по его Telegram ID. Если есть — пропускаем. Это защищает от ситуации когда нагон и модуль записи одновременно получили одно и то же сообщение. Уникальный индекс `(dialog_id, telegram_message_id)` на таблице `messages` гарантирует это на уровне БД.

**Прерванный нагон.**
Если нагон прервался на середине — при следующем старте продолжаем с последнего записанного ID. Начинать сначала не нужно — мы всегда знаем где остановились.

**Паузы между запросами.**
Нагон делает паузы между запросами к Telegram чтобы не упереться в лимиты (FloodWaitError). Конкретные значения пауз — тема симулятора человечности, закроем там.

**Работа через враппер.**
Нагон не обращается к Telegram напрямую — только через команды враппера.

---

## Обработка ошибок

- **FloodWaitError от Telegram** — ждём указанное время и продолжаем с того же места. Публикуем `system.error` для видимости в логе.
- **Ошибка записи в БД** — публикуем событие со статусом `error`, останавливаем нагон по этому диалогу. Остальные диалоги продолжаются.
- **Пропущенные события пока нагон работает** — ничего не теряется, модуль записи пишет новые сообщения параллельно, дубли отсекаются уникальным индексом.

---

## Принципы которые мы зафиксировали

1. Запускается параллельно с модулем записи истории при старте воркера
2. Сначала синхронизация списка диалогов, потом история по каждому
3. Берём максимальный кусок что отдаёт Telegram (~100 сообщений)
4. Прерванный нагон продолжается с последнего записанного ID, не сначала
5. Работает только через враппер, не обращается к Telegram напрямую
6. Ошибки публикуются на шину — `error` для событий, `system.error` для FloodWait
