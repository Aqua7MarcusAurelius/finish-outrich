# Глоссарий проекта

> Термины которые мы используем в этом проекте и что они означают.

---

| Термин | Что это |
|---|---|
| **Воркер** | Изолированный процесс для одного Telegram-аккаунта. Запускается, останавливается и управляется независимо от остальных. |
| **Враппер (Telegram Wrapper)** | Часть внутри воркера. Отвечает только за общение с Telegram. Получила событие — передала на шину. Надо отправить — отправила. |
| **Шина событий (Event Bus)** | Центральное место через которое все компоненты общаются. Никто не говорит с другими напрямую — только через шину. Является основным логом системы. |
| **Паб/Саб (Pub/Sub)** | Принцип работы шины. Кто-то публикует событие, все кто подписаны — получают его. |
| **Redis Streams** | Технология внутри Redis которую мы используем для шины событий. Хранит события по порядку, модуль не теряет события даже если был недоступен. |
| **Менеджер воркеров** | Модуль который запускает, останавливает и удаляет воркеры — без перезапуска всей системы. |
| **API** | Единая точка входа для управления системой. Принимает команды. Основной клиент — веб-дашборд. |
| **Дашборд** | Веб-интерфейс — основной клиент API. Два раздела: технический монитор и просмотр чатов. |
| **Модуль** | Независимая часть системы которая слушает шину и реагирует на нужные ей события. |
| **Событие** | Сообщение на шине. Например: "в аккаунте 1 получено сообщение от пользователя X". |
| **Telethon** | Библиотека для работы с Telegram как обычный пользователь (не бот). |
| **SOCKS5** | Тип прокси который мы используем для каждого аккаунта. |
| **PostgreSQL** | Основная база данных. Хранит всё постоянное — аккаунты, историю, настройки, архив событий. |
| **asyncpg** | Библиотека для работы с PostgreSQL напрямую через SQL. ORM не используем. |
| **Alembic** | Инструмент для управления миграциями базы данных. Создаёт и применяет изменения схемы версионно. |
| **Redis** | Быстрое хранилище. Используем для шины событий и статусов воркеров. |
| **S3** | Стандарт объектного хранилища файлов, придуманный Amazon. Сейчас S3-совместимый API поддерживают десятки провайдеров — код работает с любым без изменений. |
| **Объектное хранилище** | Хранилище где файлы лежат по ключам-строкам, а не по папкам. Доступ по HTTP. Масштабируется бесконечно. |
| **MinIO** | S3-совместимое хранилище, которое можно поднять у себя в Docker. Используем как хранилище медиафайлов. |
| **storage_key** | Ключ файла в MinIO. Формат: `{account_id}/{dialog_id}/{telegram_message_id}.{ext}`. Записан в поле `media.storage_key`. |
| **Чистильщик** | Часть модуля истории. Раз в час, пачкой до 50 записей, находит в `media` файлы старше 3 дней и удаляет их из MinIO. Метаданные в БД остаются. |
| **OpenRouter** | Прослойка между нашей системой и AI сервисами (Whisper, GPT-4o). Позволяет переключать модели без изменения кода. |
| **FFmpeg** | Инструмент для работы с видео. Используем в модуле описания медиа — нарезает кадры из видео, GIF и кружков. Работает в памяти, без временных файлов. |
| **Личный диалог** | Приватный чат между нашим аккаунтом и одним собеседником. Единственный тип диалога с которым работает система — групп и каналов не поддерживаем. |
| **is_outgoing** | Булево поле в таблицах `messages` и `reactions`. `true` = мы отправили / поставили, `false` = собеседник. Заменяет собой ссылку на таблицу пользователей, которая не нужна в личных чатах. |
| **Денормализация** | Приём когда связанные данные копируются прямо в основную таблицу, а не хранятся через ссылку на другую таблицу. Используем для пересылок: имя и id автора оригинального сообщения лежат в `messages` как снимок на момент получения. Плюс — не нужна отдельная таблица для людей которых мы видели только в пересылках. Минус — данные не обновляются если автор потом сменит имя. Для архива это как раз и нужно. |
| **Soft delete** | "Мягкое" удаление — запись не стирается физически, а получает отметку `deleted_at`. Используем для сообщений которые собеседник удалил у себя. Мы их видели, контекст разговора на них опирается, поэтому физически не стираем. |
| **Нагон** | Модуль синхронизации истории. При старте воркера догружает всё что пропустили пока он не работал. |
| **SSE (Server-Sent Events)** | Технология одностороннего потока данных от сервера к браузеру. Используем для бегущего лога, обновлений статусов воркеров и новых сообщений в чате. Браузер подключается — сервер шлёт события по мере появления. |
| **Сессия Telethon** | Данные авторизации аккаунта в Telegram в бинарном формате. Хранятся в поле `accounts.session_data` (bytea). Без сессии — нужна переавторизация. |
| **FloodWaitError** | Ошибка от Telegram о превышении лимитов запросов. Аккаунт временно блокируется, при настойчивости — навсегда. Учитывается в модуле нагона и в симуляторе человечности. |
