# Модуль транскрибации

> Переводит аудио и видео файлы из Telegram в текст для понимания контекста диалога.

---

## Вход

Модуль слушает событие `message.saved` на шине. Фильтрует по флагам:

```
if has_audio or has_video:
    → обрабатывать
```

Из payload события модуль получает:
- `media_id` — идентификатор записи в таблице `media`
- `storage_key` — ключ файла в MinIO
- `type` — тип медиа
- `mime_type`, `duration`, `file_size` — метаданные

**Важно:** модуль слушает именно `message.saved`, а не `message.received` — к моменту `message.saved` запись в `media` уже создана, можно писать результат.

---

## Выход

Модуль публикует на шину событие `transcription.done` с `media_id` и текстом результата. **Модуль транскрибации не пишет в БД сам** — обновлением `media.transcription` занимается модуль истории, слушая это событие.

Это соблюдает принцип: в каждую таблицу пишет только один модуль — тот что ей владеет.

---

## Ядро — что и чем обрабатываем

| Тип | Категория | Инструмент |
|---|---|---|
| Голосовое сообщение | Часть сообщения | OpenRouter → Whisper |
| Видео-сообщение (кружок) | Часть сообщения | OpenRouter → Whisper |
| Аудио файл | Файл | OpenRouter → Whisper |
| Видео файл | Файл | OpenRouter → Whisper (звуковая дорожка) |

> Обработка изображений, стикеров, GIF и документов — в отдельном модуле описания медиа.

### Почему OpenRouter
Прослойка между нами и AI сервисами. Если один сервис лёг или ключ отвалился — переключаемся на другую модель без изменения кода.

---

## Как получаем файл

Модуль скачивает файл из MinIO через `storage.get(storage_key)` и отправляет **байтами** в OpenRouter — не URL. Причины:
- Не зависим от того умеет ли модель читать по URL
- Байты легко логировать и проверять

Временных локальных файлов не создаём — байты живут в памяти на время запроса.

---

## Правила обработки

**Размер файла** — не ограничиваем, отправляем любой файл как есть.

**Очередь параллельности** — не используем, подаём запросы в OpenRouter как получаем события.

**Ретраи при ошибке OpenRouter** — 1 повторная попытка. Если снова ошибка — публикуем `transcription.done` со статусом `failed`. Количество попыток настраивается в `settings.transcription.retries`.

**Пустой результат Whisper** — если вернул пустой текст (тишина или неразборчиво) — делаем ещё одну попытку. Если снова пусто — записываем пустой результат со статусом `done`.

---

## Поток работы

```
[Шина: message.saved с has_audio=true или has_video=true]
    │
    ▼
[Модуль транскрибации]
    ├── читает media_id и storage_key из события
    ├── storage.get(storage_key) → байты файла
    ├── публикует transcription.started (parent_id → message.saved)
    ├── отправляет байты в OpenRouter → Whisper
    │
    ├── ошибка OpenRouter → 1 повторная попытка → снова ошибка → transcription.done (failed)
    ├── пустой результат → 1 повторная попытка → снова пусто → transcription.done (done, текст пустой)
    │
    └── публикует transcription.done (parent_id → transcription.started, media_id + текст)
         │
         ▼
    [Модуль истории]
    └── находит запись по media_id, пишет текст в media.transcription, ставит transcription_status = done
```

---

## Повторный запуск транскрибации

Через API endpoint `POST /media/{id}/retranscribe`:
- Сбрасывает `transcription_status` в `pending`
- Публикует `media.reprocess.requested` на шину
- Модуль подхватывает и проходит флоу заново

---

## Обработка ошибок

Все ошибки публикуются на шину:
- Ошибка OpenRouter, файл не найден в MinIO — событие `transcription.done` со статусом `error`
- Инфраструктурные (MinIO недоступен целиком, Redis) — `system.error`

---

## Принципы которые мы зафиксировали

1. Модуль слушает `message.saved` — к этому моменту запись в `media` гарантированно есть
2. Фильтрация по флагам `has_audio` / `has_video` — без похода в БД
3. Файл забираем из MinIO байтами, отправляем в OpenRouter тоже байтами
4. В БД модуль не пишет сам — публикует результат на шину, история дописывает в `media`
5. Модуль отвечает только за звук — картинки и документы в отдельном модуле описания
6. Размер файла не ограничиваем
7. Ретраи: 1 попытка при ошибке, 1 повторная попытка при пустом результате
8. Количество ретраев настраивается в `settings.transcription.retries` без перезапуска
