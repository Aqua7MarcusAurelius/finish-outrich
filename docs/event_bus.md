# Шина событий (Event Bus)

> Центральное место через которое все компоненты системы обмениваются информацией.
> Никто не общается с другими модулями напрямую — только через шину.
> Шина является основным логом системы — отдельного логирования не нужно.

---

## Принципы работы

- Любой компонент может опубликовать событие на шину
- Любой модуль подписывается на нужные ему типы событий
- Модули не знают друг о друге — только о шине
- Все события сохраняются и видны как лог
- Модуль который был недоступен — прочитает пропущенные события когда вернётся
- **Шина только для уведомлений** — она говорит "что случилось", но не отвечает на вопросы и не возвращает данные. Если модулю нужны данные — он идёт напрямую в Postgres, MinIO или через враппер в Telegram
- **Шина — основной лог системы.** Всё что происходит публикуется туда. Отдельного логирования приложения не нужно

---

## Структура события

Каждое событие — это сообщение со стандартным набором полей. Как конверт: снаружи служебная информация, внутри содержимое.

| Поле | Что это | Пример |
|---|---|---|
| `id` | Уникальный номер события | 001 |
| `parent_id` | ID события которое породило это | 001 |
| `time` | Когда случилось | 2024-01-15 14:23:01 |
| `account` | Какой аккаунт (null для системных) | account_1 |
| `module` | Кто опубликовал событие | transcription |
| `type` | Что случилось | message.received |
| `status` | Результат | success / error / in_progress |
| `data` | Содержимое события | текст, id сообщения и тд |

## Обработка ошибок

Все ошибки публикуются на шину со статусом `error`. Никаких отдельных систем для сбора ошибок не нужно — всё видно в едином логе.

**Два типа ошибок:**

**Ошибки в рамках события** — транскрибация не удалась, файл не загрузился и тд. Публикуются как обычное событие со статусом `error` и `parent_id` на родительское событие:
```
[277] account_1 | transcription | transcription.done | error | (→ 276) OpenRouter вернул ошибку
```

**Системные ошибки** — не связаны с конкретным сообщением или диалогом. Тип `system.error`, поле `account` может быть null:
```
[047] account_1 | worker    | system.error | error | оба прокси недоступны, воркер остановлен
[048] —         | cleaner   | system.error | error | MinIO недоступен, попробуем в следующий прогон
[049] account_2 | wrapper   | system.error | error | сессия протухла, требуется переавторизация
```

### Poison messages и dead-letter

Consumer-группа (`history`, `transcription`, `description`, `archive-writer`) подтверждает событие через `XACK` только после успешной обработки. Если обработчик упал — событие не ack'ается и прочитается заново в следующем проходе. Это защищает от потери событий при временных сбоях (БД подвисла, OpenRouter ответил 500).

Но если событие падает **всегда** (битый payload, баг в обработчике), оно копилось бы в pending-list бесконечно. Чтобы очередь не зависала, действует dead-letter:

1. На каждую неудачу обработки конкретного события инкрементится счётчик в Redis (ключ `bus:retries:{group}:{stream_id}`, TTL 24 часа).
2. При достижении `DEAD_LETTER_MAX_RETRIES = 5` — публикуется `system.error` с типом `poison_message`, событие принудительно ack'ается, счётчик удаляется.
3. При успешной обработке — счётчик сбрасывается.

Пример:
```
[150] —  | bus | system.error | error | poison_message group=transcription-worker event_id=... retries=5
```

Это видно в общем логе и в `events_archive` — разбираться с конкретным кейсом можно по `event_id` из payload.

## Ключевые события с payload

### `message.received` — пришло сообщение от Telegram
Публикует враппер после получения сообщения и (если есть медиа) загрузки файла в MinIO.

```
data: {
  telegram_message_id: 10005
  telegram_user_id: 555111222  ← id собеседника (по нему находим dialog)
  is_outgoing: false            ← наше сообщение или от собеседника
  date: 2024-01-15 14:23:01
  text: null
  media: [{ storage_key: "account_1/dialog_7/10005.ogg", type: voice, ... }]
  reply_to_telegram_message_id: null
  forward_from: null            ← объект с данными пересылки если есть
}
```

### `message.saved` — сообщение записано в БД
Публикует модуль истории после записи сообщения и всех его медиа в БД. **Это ключевое событие для всех медиа-модулей** — к этому моменту запись в `media` гарантированно существует.

```
data: {
  message_id: 42              ← наш внутренний id
  telegram_message_id: 10005
  dialog_id: 7
  is_outgoing: false

  has_text: false
  has_audio: true             ← флаги типов содержимого
  has_image: false            ← позволяют модулям быстро фильтровать
  has_video: false
  has_document: false

  media: [{ media_id: 15, storage_key: "...", type: voice, ... }]
}
```

### `message.updated` — сообщение обновлено
Публикует модуль истории после обновления (транскрипция дописалась, описание дописалось, текст отредактирован).

### `transcription.done` / `description.done` — результат обработки медиа
Публикуют модули транскрибации и описания соответственно. Содержат `media_id` и текст результата. Слушает модуль истории и обновляет запись в `media`.

### `file.cleaned` — файл удалён из MinIO
Публикует чистильщик после удаления файла. Для аудита и статистики.

### `account.created` — аккаунт авторизован
Публикует модуль авторизации после успешной записи в `accounts`.

### `account.reauthorized` — сессия переавторизована
Публикует модуль авторизации после успешной перезаписи `session_data`.

### `account.session_expired` — сессия протухла
Публикует воркер когда Telegram вернул ошибку авторизации. Воркер останавливается и ждёт переавторизации.

### `account.deleted` — аккаунт удалён
Публикует менеджер воркеров после полного удаления аккаунта и его данных.

### `worker.started` / `worker.stopped` / `worker.crashed` — статусы воркера
Публикует менеджер воркеров.

### `sync.started` / `sync.dialog.done` / `sync.done` — ход нагона
Публикует модуль нагона.

### `media.reprocess.requested` — запрос на повторную обработку медиа
Публикуется API при вызове `/media/{id}/retranscribe` или `/media/{id}/redescribe`.

### `dialog.typing_observed` — собеседник печатает
Публикует враппер, получив `UpdateUserTyping` от Telegram. Payload: `{ telegram_user_id, at }`. Слушает модуль AutoChat — использует для перезапуска reply-таймера в активной сессии. Остальные модули игнорируют.

### `autochat.started` / `autochat.initial_sent` — инициация автосессии
Публикует модуль AutoChat. `started` — сессия создана (запись в `autochat_sessions`). `initial_sent` — первое сообщение ушло через враппер.

### `autochat.entered_chat` / `autochat.left_chat` — смена InChat
Публикует AutoChat при переходах `InChat=0 → 1` (после enter-таймера 15/60/120с) и обратно (по idle 3 минуты).

### `autochat.generation_requested` / `autochat.generation_done` — обращения к Opus 4.7
`generation_requested` — ушёл запрос в OpenRouter с собранным контекстом. `generation_done` — получен ответ, распарсены `<msg>`-сегменты.

### `autochat.segment_sent` — сегмент ответа отправлен
Публикует AutoChat после каждого успешно отправленного сегмента через враппер.

### `autochat.session_stopped` / `autochat.session_error` — финал сессии
`stopped` — штатная остановка через API. `session_error` — фатальная ошибка (нас заблокировали, нельзя писать, критический сбой); сессия переводится в `status=failed`.

### `system.error` — системная ошибка
Публикует любой модуль при технической ошибке не связанной с конкретным сообщением. Поле `account` может быть null если ошибка общесистемная.

## Пример цепочки событий

Так выглядит цепочка когда собеседник прислал голосовое сообщение:

```
[001] account_1 | telegram      | message.received        | success    | получено сообщение с голосовым, файл в MinIO
[002] account_1 | history       | message.saved           | success    | (→ 001) запись в media создана, has_audio=true
[003] account_1 | transcription | transcription.started   | in_progress| (→ 002) скачал из MinIO, отправил в Whisper
[004] account_1 | transcription | transcription.done      | success    | (→ 003) расшифровано: "привет как дела"
[005] account_1 | history       | message.updated         | success    | (→ 004) текст транскрипции записан в media

... через 3 дня ...

[847] account_1 | history       | file.cleaned            | success    | (→ 001) файл удалён из MinIO
```

Пример системной ошибки:
```
[049] account_2 | wrapper | system.error | error | оба прокси недоступны, воркер остановлен
[050] account_2 | worker  | system.error | error | (→ 049) воркер остановлен, требуется вмешательство
```

## Хранение

| Хранилище | Что хранит | Зачем |
|---|---|---|
| **Redis Streams** | Живые события | Быстрая доставка всем подписчикам в реальном времени |
| **PostgreSQL** (таблица `events_archive`) | Архив всех событий | Долгое хранение, фильтрация, экспорт |

Структура таблицы `events_archive` — в `database_schema.md`.

---

## Просмотр и управление через API

- **Бегущий лог** — события в реальном времени через SSE (`/events/stream`)
- **Архив с фильтрами** — по аккаунту, модулю, типу события, времени, статусу (`/events`)
- **Цепочка событий** — рекурсивное раскрытие от корня (`/events?root_id={id}`)
- **Экспорт** — JSONL / CSV выгрузка с фильтрами (`/events/export`)

При отдаче через API к событию добавляется вычисляемое поле `message` — готовая фраза на русском для отображения в логе. В БД это поле не хранится — формулировки живут в коде API, можно менять без миграций.

Цветовое выделение по модулю и статусу — реализуем в веб-интерфейсе.

## Что не используем и почему

Существуют специальные инструменты для трассировки событий (Jaeger, Zipkin, OpenTelemetry) и отдельные системы логирования (ELK, Loki). Мы их не используем — шина событий на Redis Streams с архивом в PostgreSQL покрывает все нужные задачи в едином месте.

**Единственное исключение** — stdout контейнера. Туда попадают инфраструктурные ошибки которые не могут быть опубликованы на шину (Redis сам недоступен, Postgres упал целиком). Это минимальный safety net, без структурированного логирования.

---

## Принципы которые мы зафиксировали

1. Все компоненты общаются только через шину
2. Шина только для уведомлений — не для запросов данных
3. Каждое событие имеет `parent_id` — цепочка прослеживается
4. Все ошибки публикуются на шину — `error` для событий, `system.error` для инфраструктуры
5. Шина является основным логом системы — отдельного логирования не ведём
6. Живые события в Redis Streams, архив в PostgreSQL (`events_archive`)
7. Поле `message` для UI вычисляется на лету, в БД не хранится
