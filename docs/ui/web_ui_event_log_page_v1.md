# Страница «Event log»

> Живой и исторический лог всех событий шины.  
> Одно место для отладки, аудита и понимания что происходит в системе.

---

## Источники данных

Страница работает с двумя источниками одновременно:

| Источник | Что даёт | Роль |
|---|---|---|
| PostgreSQL (архив шины) | события любой давности | основа для фильтрации и экспорта |
| Redis Streams | живой поток в реальном времени | дописывается в конец лога на экране |

Интерфейс читает архив через API и подписывается на живой поток через SSE (или WebSocket). Когда приходит новое событие — оно появляется сверху списка без перезагрузки.

## Структура экрана

Четыре зоны сверху вниз:

```
┌────────────────────────────────────────────────────────────┐

├────────────────────────────────────────────────────────────┤

```

## Фильтры

Все фильтры применяются и к архиву (для листания назад), и к живому потоку (фильтрация на стороне сервера — в интерфейс прилетают только события под текущую выборку).

| Фильтр | Значения |
|---|---|
| Account | все аккаунты / конкретный аккаунт (из `accounts`) |
| Module | все модули / один из: telegram, history, transcription, description, auth, worker_manager |
| Event type | все типы / конкретный тип (`message.received`, `message.saved`, и т.д.) или префикс (`transcription.*`) |
| Status | любой / success / error / in_progress |
| Range | последний час / сегодня / 24 часа / 7 дней / произвольный период |
| Clear | сбрасывает все фильтры к значениям по умолчанию |

Фильтры комбинируются через `AND`. Пустой фильтр = «все».

**Range по умолчанию — последний час.** Это важно: без ограничения по времени лог моментально вырастает до миллионов записей. По умолчанию интерфейс всегда показывает окно, даже если пользователь ничего не выбрал.

## Метрики

Ряд из пяти карточек. Считаются по текущему окну фильтра.

| Метрика | Что считает |
|---|---|
| Всего за час (или за выбранный период) | `count(*)` по текущим фильтрам |
| Success | `count where status = 'success'` |
| Error | `count where status = 'error'` — подсвечена красным если не ноль |
| In progress | `count where status = 'in_progress'` |
| Events / sec | скользящее среднее за последние 60 секунд |

Задача метрик — с первого взгляда понять: всё ок или где-то посыпалось. Детали — ниже в логе.

## Колонки строки события

Плотная таблица, моноширинный шрифт для технических полей.

| Колонка | Ширина | Что показывает |
|---|---|---|
| `#id` | 70px | `events.id`, серым |
| статус | 22px | цветная точка (зелёная / красная / жёлтая) |
| время | 90px | `HH:MM:SS`, серым |
| аккаунт | 130px | `account_N` или имя аккаунта |
| модуль | 170px | цветная пилюля, фон и текст из ramp модуля |
| parent_id | 60px | `→ #N` или тире если событие корневое |
| тип + сообщение | остальное | `event.type` жирным + короткое описание из `data` |

Ошибочные события (`status = error`) целиком подсвечиваются красноватым оттенком — чтобы глазом выхватить даже при быстрой прокрутке.

## Follow tail

Чекбокс включён по умолчанию. Пока он включён — лента автоматически прокручивается вниз вверху списка к новым событиям. Как только пользователь руками прокрутил назад — чекбокс снимается автоматически, чтобы лента не «уезжала из-под пальцев» пока читают старое.

Включить обратно — кликом по чекбоксу или клавишей `End`.

## Новые события — визуальная подача

Каждая новая строка приходящая из Redis Streams:
1. Вставляется сверху списка
2. На 0.8 секунды подсвечивается голубым фоном
3. Фон плавно растворяется

Это даёт ощущение живого потока без мигания всей страницы.

## Экспорт

Две кнопки в шапке: **Export CSV** и **Export JSON**.

Экспорт работает с **текущими фильтрами**. Не со всей базой. Это важно и явно написано в интерфейсе — если нужен полный экспорт, сначала сбрасывают фильтры, потом жмут.

| Формат | Содержимое |
|---|---|
| CSV | одна строка = одно событие. Колонки: `id, parent_id, time, account, module, type, status, data_json`. Поле `data` сериализуется в JSON в одну ячейку. |
| JSON | массив объектов. Каждый объект — полное событие со всеми полями шины. |

Экспорт — стриминговый. Файл начинает скачиваться сразу, страница не блокируется.

## Детальный просмотр события

Клик по строке открывает поповер / боковую панель с полным представлением события:

```
┌─────────────────────────────────────────────────┐
│  Event #1277                             [×]    │
│  ─────────────────────────────────────────      │
│  status:    error                               │
│  time:      2026-04-22 14:22:51.412             │
│  account:   account_1                           │
│  module:    transcription                       │
│  type:      transcription.done                  │
│  parent_id: #1272 → #1271 → #1270               │
│  ─────────────────────────────────────────      │
│  Chain (parent_id):                             │
│    #1270  telegram        message.received      │
│    #1271  history         message.saved         │
│    #1272  transcription   transcription.started │
│    #1277  transcription   transcription.done ←  │
│  ─────────────────────────────────────────      │
│  data:                                          │
│  {                                              │
│    "media_id": 13,                              │
│    "error": "OpenRouter 502 Bad Gateway",       │
│    "retry_count": 1,                            │
│    "status": "failed"                           │
│  }                                              │
│  ─────────────────────────────────────────      │
│  [Копировать JSON]  [Открыть media_id=13]       │
└─────────────────────────────────────────────────┘
```

Что показываем в панели:
- Все служебные поля события
- **Цепочка `parent_id`** — раскрытое дерево всех предков события. Клик по любому — открыть его детально
- Полный `data` payload как форматированный JSON
- Кнопки: «Копировать JSON», «Открыть связанную сущность» (если в payload есть `message_id`, `media_id`, `dialog_id` — кнопка ведёт в страницу диалогов)

## Клик по колонке — быстрый фильтр

Клик по значению в колонке подставляет его в фильтры. Это самый частый сценарий навигации:

| Клик по | Результат |
|---|---|
| `account_1` | выставить фильтр Account = account_1 |
| `[transcription]` | Module = transcription |
| `transcription.done` | Event type = transcription.done |
| цветная точка статуса | Status = error / success / in_progress |
| `#1277` или `→ #1277` | открыть детальный просмотр события |

---

## Что не делаем на этой странице в v1

- Не редактируем и не удаляем события (шина — append only, логично не давать их трогать)
- Не публикуем события руками из интерфейса (это обход всей архитектуры — события публикуют только модули)
- Не показываем графики частоты событий — в v1 хватает пяти чисел в метриках
- Не сохраняем собственные «saved filters» между сессиями
- Не группируем события в цепочки визуально (дерево `parent_id` показываем только в детальном просмотре)

## Принципы которые мы зафиксировали

1. Страница работает одновременно с архивом (Postgres) и живым потоком (Redis Streams)
2. Все фильтры применяются и к архиву, и к живому потоку
3. Range по умолчанию — последний час, чтобы не тянуть всю базу
4. Цвет модуля и статуса — одинаковый с другими страницами интерфейса
5. Ошибочные события подсвечиваются целиком для быстрого обнаружения
6. Follow tail автоматически выключается когда пользователь скроллит назад
7. Новые события подсвечиваются на 0.8 секунды при появлении
8. Экспорт работает с текущими фильтрами и стримит файл
9. Клик по строке — полный payload и раскрытая цепочка `parent_id`
10. Клик по значению в колонке — быстрый фильтр по этому значению
11. События из интерфейса не редактируются, не удаляются и не создаются — только просмотр
