# 8.2 — UI: тост для ошибок API на Start/Stop воркера

**Дата:** 2026-04-24
**Этап:** 8.2
**Коммит:** `8200eec`

## Что сделано

Простой tost-провайдер (без внешних либ) + подключение к start/stop-хендлерам в `AccountCard` → теперь 409/5xx из `/workers/{id}/start|stop` всплывают красной плашкой снизу-справа, успешные действия — зелёной.

## Файлы

- [modules/ui_web/src/lib/toast.tsx](../../modules/ui_web/src/lib/toast.tsx) — новый. `ToastProvider` + `useToast()`. 3 тона (`error`/`success`/`info`), auto-dismiss: 4 сек для обычных, 8 сек для error.
- [modules/ui_web/src/lib/api.ts:11-31](../../modules/ui_web/src/lib/api.ts#L11-L31) — `describeApiError(e)` вытаскивает `{error:{code,message}}` из FastAPI-ответа в `{title, detail}`.
- [modules/ui_web/src/main.tsx](../../modules/ui_web/src/main.tsx) — обернул дерево в `<ToastProvider>` снаружи `<HashRouter>`.
- [modules/ui_web/src/pages/DialogsPage.tsx](../../modules/ui_web/src/pages/DialogsPage.tsx) — `.catch((e) => { const d = describeApiError(e); toast.error(…, d.detail); })`.

## Проблема

`AccountCard` был:
```tsx
onStart={() => api.startWorker(a.id).then(() => accountsQ.refetch()).catch(() => {})}
onStop={() =>  api.stopWorker(a.id).then(() => accountsQ.refetch()).catch(() => {})}
```

`.catch(() => {})` — тихое глотание. Когда в 8.1 мы увидели `{"error":{"code":"NOT_RUNNING"}}` с 409 статусом, в UI это выглядело как «кнопка не реагирует». Воспроизводилось на любой ошибке API, не только у воркеров.

## Решение

### Почему не sonner / react-hot-toast

Соблазн был — готовые либы выдают красивые анимации и accessibility из коробки. Решил не брать:

- Зависимость ради одного места использования — overkill.
- `sonner` тянет ~15kb, все свои стили, часто конфликтует с shadcn темами.
- Нужна только одна фича — показать строку. Без очередей, приоритетов, экшенов.

Свой `ToastProvider` — 60 строк, стили через Tailwind токены что мы уже настроили (`bg-destructive/15`, `bg-status-success/15`). Когда понадобится дашборд-сложность (группировка, приоритеты, экшены внутри тоста) — заменю на sonner.

### Почему через React Context, а не глобальный singleton

Альтернатива: `toast.error(…)` как top-level функция типа `sonner`. Работает, но:
- Глобальное состояние вне React-дерева усложняет тесты.
- Hot-reload иногда оставляет старый массив тостов висящим.
- Для текущего скоупа (один SPA, один root) context-подход проще отладить.

### describeApiError — одна точка знания о shape бэкенда

Все `ManagerError` / `HTTPException` в этом проекте возвращают `{"error":{"code":"…","message":"…"}}`. Я не хотел чтобы это знание размазалось по всем `.catch()` — вынес в `describeApiError(e)`. Хендлер пишет:
```tsx
.catch((e) => { const d = describeApiError(e); toast.error(`Stop: ${d.title}`, d.detail); })
```

И получает:
- `title`: «Воркер не запущен» (из `error.message`).
- `detail`: `NOT_RUNNING (HTTP 409)` — для более глубокой диагностики мелким шрифтом.

Когда появятся новые endpoint'ы со своим форматом — меняем один хелпер, не 10 мест.

### Тайминги

- Success: 4 сек. Пользователь видит подтверждение, не мешает дальше работать.
- Error: 8 сек. У ошибки больше текста (code + message), нужно время прочитать и понять что делать.
- Dismiss кнопкой — сразу, без ожидания таймера.

### pointer-events-none на контейнер, pointer-events-auto на тост

Контейнер тостов — `fixed bottom-3 right-3`. Без `pointer-events-none` он перекрывал бы клики по нижним элементам EventRow / MessageBubble (невидимый хит-бокс). Каждый тост явно `pointer-events-auto` чтобы крестик и hover работали.

## Проверка

1. Открыть http://localhost:3000/#/dialogs.
2. Воркер в `stopped` (после фикса 8.1).
3. Нажать Start → зелёный тост «Воркер Test account запущен».
4. Нажать Stop на том же воркере пока он стартует → красный тост «Stop: Воркер не запущен» с `NOT_RUNNING (HTTP 409)` в detail (если race'нёмся).
5. Отключить backend (`docker compose stop app`), нажать Start → красный тост с fallback-сообщением `Failed to fetch` / `HTTP 000`.

## Открытые вопросы

- **Не только для Start/Stop.** Сейчас только эти два хендлера обёрнуты. Фичи которые UI будет добавлять (`delete account`, `retranscribe media`, send-message через UI когда разрешим) должны также юзать toast. Хорошо бы к каждому мутирующему запросу автоматически — но пока не хочу делать это магическим wrapper'ом в `request()`, потому что не все вызовы мутации (некоторые side-чтения могут 404-ить нормально).
- **Очередь ≥ 5 тостов.** Пока нет лимита — если накидать ошибок быстро, столбик растёт. На практике редко бывает больше 1-2 одновременно, но для защиты можно было бы обрезать до 5.
- **Keyboard accessibility.** Нет `role="alert"` / aria-live. Для internal dev-панели некритично, но правильно было бы добавить.
- **Стрим-ошибки (SSE).** При разрыве `/events/stream` тоже хотелось бы уведомлять. Сейчас сидит как `connected: false` в хуке и нигде не рендерится. На отдельный заход.
