# 8.7 — Новый аккаунт: "+"-карточка + wizard поверх /auth

**Дата:** 2026-04-24
**Этап:** 8.7
**Коммит:** `e717f93`

## Что сделано

В ряду аккаунтов (шапка страницы «Диалоги») появилась плюс-карточка. Клик открывает трёхшаговый wizard, который по существующим `/auth/*` ручкам проводит flow: контакт-данные → код из TG/SMS → облачный 2FA-пароль (если включён).

После успеха `/accounts` рефетчится — новая карточка аккаунта появляется на своём месте. На отмену шаг ушедший в auth-state дочищается через `DELETE /auth/{session_id}`.

## Почему так

### Переиспользуем существующие /auth/* ручки

Backend уже умел весь flow (`modules/auth/routes.py` + `modules/auth/service.py`) — раньше им пользовались руками через curl / Postman. UI ничего нового в backend не тащит, только оборачивает в формы. См. маппинг шагов:

| Шаг UI | Ручка | Что приходит обратно |
|---|---|---|
| 1 (form)  | `POST /auth/start` | `{session_id, status:"code_sent"}` |
| 2 (code)  | `POST /auth/code`  | `{status:"2fa_required"}` или `{status:"done", account_id}` |
| 3 (2fa)   | `POST /auth/2fa`   | `{status:"done", account_id}` |
| отмена    | `DELETE /auth/{session_id}` | 204 |

Статусы — это строки `PHASE_*` из [modules/auth/service.py:42-45](../../modules/auth/service.py#L42-L45).

### Machine — в компоненте, не в Redux / Zustand

Три шага, локальное состояние, нет разделяемой логики с другими страницами. Классический кейс для `useState` — без стор-библиотеки. Когда появится что-то кроссстраничное (список активных auth-сессий, мониторинг progress'а reauth) — вынесу в store.

### Отмена best-effort, не блокирующая

При закрытии модалки `api.authCancel(session_id).catch(() => {})`. Если session_id ещё нет (пользователь закрыл до step 1) или cancel упал — UI не застревает. На backend оставшуюся висящую сессию добьёт TTL (см. `_write_state` TTL в auth/service).

Не делал синхронного ожидания `await` — пользователь не должен смотреть спиннер на закрытии модалки.

### Плюс-карточка — `border-dashed` вместо сплошной

Визуально отделяет «это не аккаунт, это action» от настоящих аккаунтов. Hover — border становится более контрастным + текст белеет. Ширина `min-w-[140px]` — уже чем реальные карточки (220px), чтобы в ряду она воспринималась как дополнение, а не «ещё один воркер».

### Поля формы — именно те 4 что требует `AuthStartIn`

Схема pydantic-а:
```python
class AuthStartIn(BaseModel):
    phone: str = Field(min_length=5, max_length=20)
    name: str = Field(min_length=1, max_length=200)
    proxy_primary: str
    proxy_fallback: str
```

В UI все 4 обязательны (`formValid` требует их непустоты). Прокси — mono-шрифт, чтобы `socks5://user:pass@host:port` было читаемо. Валидация формата прокси на фронте не делаю: backend всё равно перепроверит при первом connect'е воркера, плюс форматов несколько (socks5/http) — не хочется ложных блокировок.

### Без внешних form-библиотек (react-hook-form / formik)

4 поля, 3 шага, без сложной валидации — каждый `useState` обозримо. `react-hook-form` был бы overkill и +15kb бандла.

### Успех — тост + refetch + reset + close

Порядок действий в `submitCode`/`submit2fa` при `status=done`:
1. `toast.success("Аккаунт добавлен", account_id=N)` — пользователь видит подтверждение.
2. `reset()` — очищаем форму (если wizard откроется снова, он пустой).
3. `onCreated()` → `accountsQ.refetch()` — новая карточка появляется.
4. `onClose()` — закрывает модалку.

Именно в такой последовательности: `reset` до `close`, чтобы анимация закрытия не видела пустые поля.

## Проверка

1. Открыть http://localhost:3000/#/dialogs — в ряду карточек справа видна «Новый аккаунт» с плюсом.
2. Клик → модалка с полями phone / name / proxy_primary / proxy_fallback.
3. Жмёшь «Отправить код» — если прокси невалидный, тост с ошибкой backend'а (`Auth start: …`).
4. Если ок — переход на шаг 2, текст «Код отправлен в Telegram / SMS на +...». Ввод кода → либо успех, либо переход на шаг 3 (если аккаунт с 2FA).
5. На любом шаге кнопка «Отмена» закрывает модалку и чистит session.

## Открытые вопросы

- **Прогресс-индикатор во время long-running операций.** `submitForm` делает реальный connect к Telegram через `start()` — может занимать 2-5 секунд. Сейчас просто disable+"…" на кнопке. Пойдёт для MVP, позже можно добавить полноценный `<Spinner>` с «соединяемся с Telegram…».
- **Валидация phone на фронте.** Backend принимает min 5 / max 20. Проверял только длину. Можно добавить regex `^\+?[0-9]{5,20}$` чтобы не улетало в backend мусор вроде букв.
- **Ретрай кода.** Сейчас если ввёл неверный код — сессия в backend'е остаётся в `PHASE_CODE_SENT`, можно попробовать снова. Но если backend мягко блокирует после N попыток, UI обработает как обычную ошибку. Хорошо бы явно показать «попытки: 2/3». Не сейчас.
- **Переавторизация (`/auth/reauth`).** В backend есть ручка для reauth'а существующего аккаунта при session_expired. В UI пока Reauth-кнопка на AccountCard просто дёргает `startWorker` (что почти наверняка сломается). Подключу reauth отдельным этапом, когда увидим живой session_expired.
- **Список висящих auth-сессий.** Если пользователь начал wizard, закрыл вкладку без клика «Отмена» — сессия висит до TTL. Не проблема, но теоретически видно через `GET /auth/status/{session_id}` если запомнить последний.
