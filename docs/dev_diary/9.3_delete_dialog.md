# 9.3 — Жёсткое удаление диалога (X в шапке чата)

**Дата:** 2026-04-26
**Этап:** 9.3

## Что сделано

Добавил красную кнопку X в шапку чата на странице диалогов. Клик → confirm-
модалка → `DELETE /dialogs/{id}`. После удаления собеседник для системы
становится «новым»: следующее сообщение от/к нему создаст диалог с пустой
историей.

Под капотом: останавливается активная autochat-сессия (если есть), сносятся
MinIO-файлы media, и cascade DELETE выносит messages → media/reactions/
message_edits.

## Файлы

### Backend
- `core/events.py` — `EventType.DIALOG_DELETED = "dialog.deleted"`.
- `modules/history/routes.py` — `DELETE /dialogs/{id}`. Шаги: lookup
  dialog + active autochat session + storage_keys; stop_session через
  `app.state.autochat_service`; remove_object для каждого MinIO-файла;
  DELETE FROM dialogs (cascade); publish `dialog.deleted` с метаданными
  для аудита.

### Frontend
- `modules/ui_web/src/lib/api.ts` — `deleteDialog(dialogId)`.
- `modules/ui_web/src/components/dialogs/DeleteDialogConfirm.tsx` —
  модалка подтверждения с красной destructive-кнопкой и списком
  что именно будет удалено.
- `modules/ui_web/src/pages/DialogsPage.tsx` — иконка X в правом
  углу шапки чата (видна только когда выбран диалог), стейт +
  обработчик `confirmDelete`, рефетч списка диалогов, redirect на
  `/dialogs/{accountId}` после успеха.

## Решения

### CASCADE на FK, не ручное удаление
В миграции 0001 уже стоит `ON DELETE CASCADE` на `messages.dialog_id` и
далее по цепочке `media`/`reactions`/`message_edits` от
`messages.id`. Один `DELETE FROM dialogs WHERE id=$1` — и всё связанное
уходит атомарно. Не пишем явных DELETE по каждой таблице.

### Активная autochat-сессия — стопится явно перед DELETE
`autochat_sessions.dialog_id` имеет `ON DELETE SET NULL`. Если
просто удалить диалог, активная сессия останется живой, но с
`dialog_id=NULL` — и ляжет на следующем `_generate_and_enqueue` (там
проверка `if self.dialog_id is None: return`). Чтобы этот
side-effect не оставлять висеть, явно зовём `stop_session()` через
`app.state.autochat_service` до удаления. Если stop падает —
ошибку глотаем (CASCADE NULL всё равно отработает).

### MinIO-файлы — удаляем сразу, без ожидания cleaner'а
Cleaner работает по TTL (3 дня по умолчанию). Если оператор удалил
диалог чтобы сделать собеседника «новым», ждать 3 дня прежде чем
файлы реально пропадут из storage — странно. Удаляем сразу, ошибки
не блокируют главный flow (orphan-файл в MinIO не страшен — cleaner
его не подхватит, потому что нет media-row в БД, но и не помешает).

### events_archive не трогаем
Это исторический лог по архитектуре проекта. dialog_id внутри
JSONB `data` останется в старых событиях — это валидное поведение
audit-trail. Если понадобится — в UI можно фильтровать события по
существующим dialog_id.

### Confirm-модалка обязательна
Действие необратимо и сносит данные пользователя. Per
session-инструкции — destructive operations всегда подтверждаются.
Модалка явно перечисляет что удалится (сообщения / медиа /
авто-сессия / файлы) и предупреждает большим красным текстом
«Действие необратимо». Кнопка "Удалить" — destructive variant.

### 204 No Content на success
Стандарт REST для DELETE без полезной нагрузки. UI игнорирует body,
проверяет только статус.

## Аудит-событие

```json
{
  "module": "api",
  "type": "dialog.deleted",
  "status": "success",
  "account_id": 1,
  "data": {
    "dialog_id": 95,
    "telegram_user_id": 7576197923,
    "username": "vians777",
    "media_files_attempted": 0,
    "stopped_autochat_session_id": null
  }
}
```

`media_files_attempted` — сколько MinIO-файлов пытались снести (без
учёта успеха каждого). `stopped_autochat_session_id` — ID активной
autochat-сессии что мы остановили (null если не было).

## Что НЕ сделано

- Soft-delete с возможностью восстановления — намеренно: оператор
  выбирает "сделать собеседника новым", это окончательное действие.
- Bulk-delete нескольких диалогов разом — пока не нужно.
- Подтверждение через ввод username/имени — overkill для нашего
  scope. Простая красная кнопка справляется.
