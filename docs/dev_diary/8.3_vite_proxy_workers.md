# 8.3 — Vite proxy: добавил /workers (а заодно /auth, /autochat)

**Дата:** 2026-04-24
**Этап:** 8.3
**Коммит:** `2e81703`

## Что сделано

Расширил список префиксов в `vite.config.ts:proxy` до полного покрытия роутеров из `api/main.py`. Без этого Start/Stop в AccountCard возвращали 404 потому что Vite не знал что с `/workers/*` делать.

## Файлы

- [modules/ui_web/vite.config.ts:28-40](../../modules/ui_web/vite.config.ts#L28-L40) — добавлены `/workers`, `/auth`, `/autochat`.

## Проблема

UI жмёт Start → `api.startWorker(1)` → `fetch("/workers/1/start", {method:"POST"})`. Браузер отправляет POST на `http://localhost:3000/workers/1/start`. Vite-dev-server получает его, ищет правило в `server.proxy` — нет `/workers` в списке. Попадает под общий handler, который:
- для GET — отдал бы index.html (SPA fallback),
- для POST/DELETE — возвращает **404** (SPA-fallback не применим).

Тост из 8.2 это честно показал: `HTTP 404` / `Start: HTTP 404`. Бэкенд при этом был абсолютно рабочий — `curl -X POST :8000/workers/1/start` возвращал `200 {"status":"starting"}`.

## Решение

Одно место — список `proxy` в `vite.config.ts`. Было 7 префиксов, стало 10:

```ts
"/workers":  { target: apiTarget, changeOrigin: true },
"/auth":     { target: apiTarget, changeOrigin: true },
"/autochat": { target: apiTarget, changeOrigin: true },
```

## Почему так, а не иначе

### Почему не одно общее правило вроде `/api/*`

Соблазн — сделать FastAPI под префиксом `/api` (через `app.include_router(…, prefix="/api")`) и прокси `"/api"` закрывал бы всё. Не взял:

- Потребовалось бы пройти по всем документам (`docs/api.md`, `docs/ui/web_ui_api_contract_v1.md`) и переписать пути.
- Сломались бы все закладки / curl-команды / внешние клиенты которые уже знают про `:8000/events`, `:8000/workers`.
- Автодок `/docs` у FastAPI тоже пришлось бы переехать.

Менять `vite.config.ts` один раз, когда добавляется новый роутер — дешевле.

### Почему явный список, а не catch-all `"/"`

Альтернатива — `proxy: { "/": … }` с исключениями для `/src`, `/@vite`, `/node_modules`. Технически работает, но:
- HMR-клиент Vite живёт на `/@vite/client` и использует WebSocket. Тонко настраивать исключения под это хрупко.
- SPA-пути сейчас после `#` (HashRouter из 8.2), но если когда-то вернёмся на BrowserRouter — catch-all всё сломает.

Явный список + комментарий «соответствует api/main.py» — самоочевидно что делать при новом роутере.

### /auth и /autochat добавил впрок

Пока UI в них не ходит. Но точно будет — модуль авторизации новых аккаунтов и управление AutoChat-сессиями в roadmap. Добавил сейчас чтобы не натыкаться на ту же ловушку дважды.

### Почему в dev-mode это вообще проблема, а в prod — нет

В prod поднимается reverse-proxy (nginx), в конфиге которого все пути честно расписаны. Vite dev-proxy — фиктивная замена nginx на время разработки, и он явно требует `server.proxy: { "<prefix>": ... }` — без этого любой путь который выглядит как API становится 404.

Чтобы не попасть снова, в комментарии к `proxy` прямо написал: «Каждый префикс должен соответствовать реальному роутеру в api/main.py».

## Про вопрос пользователя «покажи все воркеры»

В БД сейчас **один** аккаунт — `Test account (+447350135778)`. Карточки показывают ровно то что есть:
```sql
SELECT id, name, phone, is_active FROM accounts ORDER BY id;
 1 | Test account | +447350135778 | t
```

Добавить новые аккаунты через UI пока нельзя — в спеке [docs/ui/web_ui_overview_v1.md](../ui/web_ui_overview_v1.md) это помечено «⏸ Отложена». Пока через API: `POST /auth/send-code` → `/auth/verify-code` → `/auth/complete-2fa` (см. `modules/auth/routes.py`).

Когда UI-wizard понадобится — сделаю отдельным этапом.

## Проверка

```
$ curl -sS -X POST http://localhost:3000/workers/1/start
{"account_id":1,"status":"starting"}

$ curl -sS http://localhost:3000/accounts
[{"id":1, ..., "status":"running", "last_event_at":"2026-04-24T14:46:26..."}]

$ curl -sS -X POST http://localhost:3000/workers/1/stop
{"account_id":1,"status":"stopping"}
```

В UI: Start → зелёный тост «Воркер Test account запущен», точка в карточке зелёная, кнопка становится Stop. Обратно — зеркально.

## Открытые вопросы

- **UI для добавления аккаунта.** Выше. В roadmap.
- **Обновление карточки в realtime.** Сейчас `useAccounts` polling'ит раз в 5сек (`refetchInterval: 5_000`). Пока воркер стартует, ~1-2 сек визуально «stopped». Можно подписаться на `GET /workers/stream` (SSE от worker_manager) и обновлять мгновенно. На отдельный заход.
- **Прогресс запуска воркера.** При Start воркер проходит через `starting` → `running`. UI сейчас рендерит обе стадии одинаково (Stop-кнопкой). Было бы честнее показывать `starting` как disabled/loading. Мелочь, можно позже.
