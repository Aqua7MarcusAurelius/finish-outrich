# 9.1 — Превью промта на странице редактора ("Что уйдёт в LLM")

**Дата:** 2026-04-26
**Этап:** 9.1 (после 9.0 — структурированный редактор промтов)

## Что сделано

Добавил кнопку **"Что уйдёт в LLM"** в футер страницы `/workers/{id}/prompt`.
Открывает модалку с собранным system+user-промтом ровно в том виде, в
котором он уйдёт в `chat_completion` — без вызова LLM и без сохранения
полей в БД. Удобно итерировать промт без живого Telegram'а и без расхода
токенов.

История подставляется из реального диалога этого воркера. По умолчанию —
случайный диалог из списка (auto-pick при первом открытии модалки), оператор
может переключить через dropdown или выбрать "(без истории)" чтобы видеть
чистую структуру промта.

## Файлы

### Backend
- `api/routes/accounts.py` — `POST /accounts/{id}/prompts/preview`. Body =
  8 reply-полей + опц. `dialog_id` + опц. `user_system_prompt`. Возвращает
  `{system, user, dialog_id}`. Внутри переиспользует `render_reply_system`
  и `build_conversation_context` (без сохранения, без обращения к LLM).

### Frontend
- `modules/ui_web/src/components/dialogs/PromptPreviewDialog.tsx` — новая
  модалка. Dropdown диалогов, два моно-pre с system/user, кнопка обновить +
  кнопка скопировать. Auto-pick случайного диалога при открытии.
- `modules/ui_web/src/lib/api.ts` — `previewWorkerPrompts()`.
- `modules/ui_web/src/pages/WorkerPromptPage.tsx` — кнопка "Что уйдёт в
  LLM" в футере (слева, отделена от Save/Cancel через flex-1 spacer).

## Решения и почему именно так

### Не вызываем LLM в превью
Это превью промта, а не реальный прогон. Реальный прогон — отдельная фича
("Прогон" с настоящим chat_completion), пока не нужна. Превью даёт
оператору 95% полезного — увидеть итог сборки и поймать "ой, заметка
про собеседника пустая" / "ой, конфиг отправит длинную историю".

### Случайный диалог по умолчанию
Изначально дефолт был "(без истории)" — оператор должен был сам выбрать.
Это лишний клик. Случайный показывает живые данные сразу, оператор сразу
видит как промт выглядит на реальной переписке. Если хочется чистой
структуры — есть опция в dropdown.

Auto-pick срабатывает один раз за сессию модалки (флаг `dialogPicked`).
Защита от повторного выбора: если оператор кликнул "(без истории)", и
потом сработал рефетч `useDialogs` — мы НЕ перевыбираем за оператора.
Закрыл и открыл — снова случайный.

### По кнопке "Обновить", не на каждый keystroke
Live-перерисовка на каждый keystroke = N запросов в секунду на бэк.
Оператор сам решает когда смотреть итог (отредактировал поле → нажал
обновить).

### Не передаём `initial_system` в превью
Превью только для reply-промта. Initial — отдельная история, рендерится
другим путём (`build_initial_messages`), оператор увидит его эффект
только при реальном `/autochat/start`.

### Эндпоинт принимает поля из тела, не читает БД
Это критично — оператор может посмотреть превью НЕсохранённых полей.
Если бы эндпоинт читал из `account_prompts`, превью бы показывало
последнее сохранённое, а не текущее в textarea. Бесполезно для итерации.

### Не валидируем что dialog_id принадлежит account_id… а валидируем
Сначала хотел не валидировать (превью же, безопасный read-only). Потом
понял что без валидации можно через preview прочитать историю чужого
диалога (data leak). Добавил проверку `WHERE id=$1 AND account_id=$2`.
Возвращает 400 `DIALOG_NOT_FOUND`.

### `_render` импортируется из generation.py (private name)
Используется для безист-режима (без dialog_id) — нужно подставить
`{conversation_history}` placeholder руками. Альтернатива — добавить
`history_text` параметр в `build_conversation_context`. Перебор для одного
кейса. Импорт `_render` пометил как pragmatic — Python не enforce'ит
private, функция стабильная.

## Sanity-check

```
$ curl -X POST /accounts/1/prompts/preview -d '{"fabula":"Тест","bio":"Я Маша",...,"dialog_id":95,"user_system_prompt":"новый собеседник"}'
{
  "system": "# Контекст задачи\n\nТест\n\n# Кто ты\n\nЯ Маша\n\n# Контекст\n\nТекущее время: 26.04.2026 09:51:33\n\nЗаметка про собеседника: новый собеседник\n\n# История переписки\n\nAuthor: Ты\nDate: 24.04.2026\nWeekday: Пятница\nTime: 10:08:54\nText: привет) разгребаю...",
  "user": "Ответь сейчас.",
  "dialog_id": 95
}
```

Без `dialog_id` — `# История переписки` секция содержит placeholder-маркер.

## Открытые вопросы

- Кнопка "Сохранить и протестить через LLM" — реальный chat_completion +
  показ ответа модели. Уже не превью, отдельная фича. Когда оператор
  захочет тюнить итеративно — добавим.
- Live-перерисовка по debounce (не на каждый keystroke, но на пауза
  500мс) — рассмотрим если ручной "Обновить" окажется неудобным.
