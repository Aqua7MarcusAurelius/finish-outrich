**Дата:** 2026-04-24
**Этап:** 8.0

## Что сделано

Новый модуль `modules/ui_web/` — read-only веб-панель поверх FastAPI по спецификации из [docs/ui/](../ui/). Stack: Vite + React 19 + TS + Tailwind + shadcn/ui, упакован в отдельный docker-compose сервис `ui` на порту 3000. Проксирует API-запросы через Vite dev-server на FastAPI (`app:8000` в Docker-сети).

Две страницы: **Диалоги** (аккаунты + список диалогов + сообщения с медиа/транскрипциями/описаниями/реакциями/forward/reply/soft-delete) и **Event log** (архив + SSE live-поток + 5 метрик + фильтры + follow-tail + модалка цепочки `parent_id`). Пока backend не докатил конкретную ручку — UI показывает `ErrorBox` в соответствующей секции и не валится.

## Почему так, а не иначе

### Отдельный модуль, а не встроенные в FastAPI статики

UI — read-only тонкий слой поверх API (см. [docs/ui/web_ui_overview_v1.md](../ui/web_ui_overview_v1.md)). Держать его в отдельном контейнере:
- падение UI не трогает воркеры и шину;
- разные зависимости (Node 22 vs Python 3.12) не перемешиваются;
- фронтенд-разработчику дают `docker compose up ui` и всё.

Это ровно тот же принцип изоляции, по которому живут `transcription`, `description`, `autochat`.

### Vite + React + shadcn/ui, не Next.js / SvelteKit

SSR не нужен (внутренняя панель, ~10 пользователей). Next.js тащил бы лишний сервер и сложности с прокси. Vite минималистичнее, а shadcn/ui — исходники под рукой, проще кастомизировать под «плоский плотный интерфейс» из спеки. shadcn-примитивы (Button, Card, Badge, Input, ScrollArea, Select, Dialog) положены руками в `components/ui/` — CLI не запускал, чтобы не зависеть от сетевого npx.

### Backend: адаптеры в UI вместо переименований полей

Стандартный сериализатор `_message_to_dict` в `modules/history/routes.py` возвращает `forward_from`, `reply_to_message_id`, `file_size` — это формат с prev.стадий, его читают `autochat`, `worker/wrapper`, `history/service`. Переименовать под контракт UI означало пройтись по 14+ файлам и сломать горизонтальную шину. Вместо этого `src/lib/api.ts` маппит shape → `forward`, `reply_to`, `size_bytes`. Переименование backend'а = отдельная задача, когда все консьюмеры перейдут на новый shape.

### Курсорная пагинация через base64(JSON)

Для больших таблиц (events, messages) offset медленнее с каждой страницей. Курсор — непрозрачная строка с парой `(time, id)`, сервер делает `WHERE (time, id) < (cursor_time, cursor_id)`. Помогает B-Tree индексу. Клиент передаёт курсор обратно как есть.

### SSE, а не WebSocket для live-потока

Поток односторонний (backend → UI). SSE работает через обычный HTTP, автоматический реконнект встроен в браузер, проще проксируется через nginx. WebSocket был бы перебором.

### Фильтрация SSE — на сервере

В `/events/stream` UI передаёт `account`, `module`, `type`, `status`. Сервер фильтрует в `_match_filters_memory()` — повторяет семантику `_build_filter_sql()`. Клиент ничего не фильтрует у себя. При смене фильтров UI пересоздаёт `EventSource` (useEffect на 4 ключа).

### HashRouter, не BrowserRouter

Плата за то что UI и API делят один origin. `/events` — и React-роут страницы, и proxy на FastAPI-ручку. Клик по NavLink перехватывается клиентом, но F5 делал реальный GET и отдавал сырой JSON. Альтернативы:
- префиксовать SPA под `/ui/*` — 15 точек правок, 15 шансов забыть;
- префиксовать API под `/api/*` — пришлось бы трогать FastAPI и все консьюмеры;
- HashRouter — одна строка, URL после `#` на сервер вообще не уходит.

Взял HashRouter. Для внутренней dev-панели `/#/events` — нормально.

### Polling для Vite HMR

Bind-mount Docker на Windows не пробрасывает inotify. Без `server.watch.usePolling: true` каждая правка requires ручной `docker compose restart ui`. 300мс polling — чуть больше CPU, но HMR работает.

### Цвета модулей в CSS-переменных

Каждый модуль шины имеет постоянный цвет на всех страницах (спека). CSS vars в `src/index.css`, Tailwind читает их через `hsl(var(--module-wrapper))`. Расширение списка модулей — править две точки (`index.css` + `tailwind.config.ts`), плюс Tailwind-JIT автоматически собирает классы `bg-module-wrapper/20`. `ModulePill` с fallback на neutral-серый для неизвестных модулей — новый модуль шины не ломает строку.

### BusEvent.id — `string`, не `number`

Одна из двух причин наезжания колонок в первом подходе — типовая ошибка. В `core/bus.py` id это 32-символьный hex UUID, а мой первый типовой слой имел `number`. Столбец `#id` на 70px не помещал его и overflow бил соседние ячейки. Исправил тип сквозь api.ts/types/EventLogPage/EventDetailDialog и начал рендерить `#abcdef12` — первые 8 hex'ов, полный id в `title`.

### min-w-0 на каждом grid-child

Вторая причина наезжания. CSS grid дефолтит `min-width: auto = content size` — длинный текст игнорирует `grid-cols-[70px ...]`. Нужно явное `min-w-0 truncate` на каждой ячейке + `minmax(0,1fr)` на последней. Общая ловушка Tailwind grid, всплывает постоянно.

## Открытые вопросы

- **`account_name` в событиях.** Backend не отдаёт `account_name` в `/events` и SSE — UI показывает fallback `account_${id}`. Добавить в `_enrich()` LEFT JOIN на accounts и положить поле в payload.
- **Reactions ungrouping.** UI группирует реакции клиентом в `adaptMessage()`. Логичнее сразу отдать сгруппированный массив из backend (один SQL с `GROUP BY emoji`). Пока так, нагрузки маленькие.
- **`reply_to.text_preview` не приходит.** Контракт ожидает превью текста оригинала в reply — backend возвращает только `reply_to_message_id` + `reply_to_telegram_message_id`. Адаптер в UI ставит `text_preview: null`. Подтянуть превью в backend (`_fetch_reply_previews` заготовка уже есть в `modules/history/routes.py`).
- **Пагинация диалогов.** `/accounts/{id}/dialogs` пока отдаёт до 200 строк и не даёт курсора. Для тестовых аккаунтов хватает, для прода (тысячи диалогов) нужна курсорная.
- **`/events/export` потолок 100k строк.** Жёсткий лимит чтобы архив в миллионы событий не забил процесс. Если фильтром реально нужен больший объём — добавить параметр или переключиться на chunked-ответ.
- **Добавление аккаунтов из UI.** В v1 помечено как «отложено». Сейчас auth-flow только через API — в `POST /auth/...`. UI-wizard добавим, когда понадобится.
