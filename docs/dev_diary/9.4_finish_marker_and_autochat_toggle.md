# 9.4 — Маркер `<finishdialog/>` + toggle автодиалога в шапке чата

**Дата:** 2026-04-26
**Этап:** 9.4

## Что сделано

Две связанные фичи управления автодиалогом из UI:

1. **LLM может сама гасить сессию** — поставив маркер `<finishdialog/>`
   в свой ответ. Парсер вырезает маркер, последние сегменты улетают
   нормально, потом сессия останавливается.
2. **Кнопка-toggle "Авто: ВКЛ/ВЫКЛ"** в шапке чата рядом с X. Вкл —
   создаёт active-сессию, ждущую входящих (без отправки initial). Выкл —
   стопит активную.

## Файлы

### Backend
- `core/events.py` — `EventType.AUTOCHAT_FINISHED_BY_LLM`.
- `modules/autochat/errors.py` — `DialogNotFound` для toggle-эндпоинтов.
- `modules/autochat/generation.py` — `extract_finish_marker(response)` +
  regex `<\s*finishdialog\s*/?\s*>` (case-insensitive, толерантный к
  пробелам и `<finishdialog>` без слэша).
- `modules/autochat/session.py` — конструктор принимает
  `on_finished_by_llm` callback. После `extract_finish_marker` маркер
  `finish_after=True` ставится в payload последнего сегмента; в
  `_send_segment` после успешной отправки последнего сегмента
  публикуется `autochat.finished_by_llm` и в фоне зовётся callback,
  который дёргает `service.stop_session(self.id)`. Edge: если LLM
  прислала ТОЛЬКО маркер без сегментов — финиш триггерится сразу,
  без отправки.
- `modules/autochat/service.py` — три новых метода:
  - `status_for_dialog(dialog_id)` — есть ли активная сессия
  - `enable_for_dialog(dialog_id)` — включить (без LLM/initial),
    создаёт сессию `status='active'` сразу
  - `disable_for_dialog(dialog_id)` — выключить активную
  - `_on_session_finished_by_llm(session_id)` — обёртка над
    `stop_session` для callback'а из session.
  Все 3 места создания `AutoChatSession` (restore, create_session,
  enable_for_dialog) теперь передают callback.
- `modules/autochat/routes.py` — добавлен второй роутер
  `dialog_autochat_router` под `/dialogs/{id}/autochat` (GET/POST/DELETE).
- `api/main.py` — подключение второго роутера.

### Frontend
- `modules/ui_web/src/lib/api.ts` —
  `getDialogAutochat`/`enableDialogAutochat`/`disableDialogAutochat`.
- `modules/ui_web/src/pages/DialogsPage.tsx` — useQuery с
  `refetchInterval=5000` для статуса автодиалога. Кнопка-toggle
  слева от X с зелёной/серой точкой и иконкой `Bot`.

## Решения и почему

### Маркер `<finishdialog/>`
Выбрал XML-тег чтобы быть консистентным с `<msg>...</msg>` который
LLM уже умеет ставить. Regex принимает варианты `<finishdialog/>`,
`<finishdialog>`, с пробелами — модели вольны в форматировании,
строгий matcher отстрелит валидную попытку.

Маркер вырезается ДО `parse_segments` — иначе он мог бы попасть
внутрь последнего `<msg>` или сесть отдельным сегментом и улететь в
Telegram как видимый текст.

### Финиш — после отправки последнего сегмента, не до
Прощальный реплай должен уехать. Если погасить сессию до отправки —
последнее слово LLM пропадёт. Поэтому `finish_after=True` живёт
в payload САМОГО ПОСЛЕДНЕГО сегмента, и триггер срабатывает в
`_send_segment` уже после `wrapper.send_message` и публикации
`autochat.segment_sent`.

### Callback вместо прямой ссылки на service
В `AutoChatSession.__init__` пробрасываем `on_finished_by_llm`
callback (`Callable[[int], Any]`), а не сам сервис. Сессия не должна
знать о существовании сервиса — только дёрнуть «я закончила». Сервис
в обёртке `_on_session_finished_by_llm` делает обычный `stop_session`,
который чистит индексы, обновляет БД, публикует
`autochat.session_stopped`.

### `asyncio.create_task` для финиша
Если бы вызвали `await self._on_finished_by_llm(...)` напрямую,
получили бы дедлок: callback зовёт `stop_session`, который зовёт
`session.stop()`, который ждёт завершения `sender_task` — это таска,
из которой мы прямо сейчас делаем await. Через `create_task` —
fire-and-forget, sender возвращается из `_send_segment` и попадает
в `_send_queue.get()`, где его cancel'ит `session.stop()`.

### Toggle-сессия = `status='active'` сразу, без `'starting'`
В отличие от `create_session`, тут нет шага "вызвать LLM и отправить
сообщение". Если бы было `'starting'` — потом ещё один UPDATE на
`'active'` без видимой причины. Сразу `'active'`, чисто.

`autochat_sessions.system_prompt` и `initial_prompt` (NOT NULL колонки)
заполняются пустыми строками — мы их в новой схеме не используем.

### Polling статуса каждые 5с
LLM может погасить сессию автономно (через `<finishdialog/>`) — UI
должен это увидеть без ручного refresh. 5 секунд — баланс между
живостью UI и нагрузкой на бэк (один лёгкий запрос в БД на диалог).

### REST-семантика для toggle
- `GET /dialogs/{id}/autochat` — состояние
- `POST /dialogs/{id}/autochat` — включить
- `DELETE /dialogs/{id}/autochat` — выключить

Естественнее чем `POST /autochat/sessions/{id}/stop` для UI — клиент
оперирует диалогом, а не session_id (который ещё нужно резолвить).

### Второй роутер `dialog_autochat_router`
Существующий `autochat_router` имеет `prefix="/autochat"`. Эндпоинты
`/dialogs/{id}/autochat` нужны без этого префикса. Самое простое —
второй роутер без префикса, оба регистрируются в `main.py`. Сам сервис
один (`request.app.state.autochat_service`), общий между роутерами.

## Что НЕ сделано

- Восстановление активных сессий при рестарте уже было в
  `_restore_active_sessions` — теперь оно тоже передаёт callback,
  работает прозрачно. Финиш-маркер от LLM сработает и для
  восстановленной сессии.
- Отдельной кнопки "перезапустить с нуля (включая initial)" нет —
  если оператору это надо, удаляет диалог через X и заново через
  "+ Новый авто-диалог".

## Как использовать `<finishdialog/>`

Оператор добавляет в свой reply-промт инструкцию вроде:

> Если ВСЕ цели разговора достигнуты и больше нечего обсуждать —
> добавь в самом конце ответа маркер `<finishdialog/>` после последнего
> `<msg>`.

Пример того что вернёт LLM при достижении цели:

```
<msg>да, договорились</msg>
<msg>напишу когда что-то будет 🙂</msg>
<finishdialog/>
```

Парсер увидит маркер, вырежет, отправит два сегмента, пометит сессию
как `stopped` и опубликует `autochat.finished_by_llm`. UI через
5-секундный poll увидит что `active=false` и переключит кнопку на
"Авто: ВЫКЛ".

## Sanity-check

```bash
# Status — нет dialog
$ curl /dialogs/9999/autochat
{"error":{"code":"DIALOG_NOT_FOUND",...}} [404]

# Status — диалог есть, активной сессии нет
$ curl /dialogs/211/autochat
{"active":false,"session_id":null,"status":null}

# POST — без воркера
$ curl -X POST /dialogs/211/autochat
{"error":{"code":"WORKER_NOT_RUNNING",...}} [409]

# DELETE — стопит активную (если была)
$ curl -X DELETE /dialogs/211/autochat
{"session":{"id":4,"status":"stopped",...}}
```
