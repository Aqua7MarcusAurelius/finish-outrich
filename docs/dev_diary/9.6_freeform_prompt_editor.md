# 9.6 — Свободный редактор промта (два textarea + плейсхолдеры)

**Дата:** 2026-05-01
**Этап:** 9.6

## Что сделано

После эксперимента со структурированными 8-полевыми редакторами (9.0)
вернулись к свободному редактированию: два больших textarea — один под
initial-промт, второй под reply-промт. Внутри текста оператор расставляет
плейсхолдеры `{current_time}`, `{worker_name}`, `{partner_username}`,
`{conversation_history}` и т.д. — система подставляет реальные значения
во время генерации.

Старые 8 структурированных полей удалены вместе с данными — оператор
просил начать с чистого листа.

## Решения и почему

### Возврат к свободному формату
Структурированные 8 полей помогали "не забыть" критичные части (формат,
запреты), но искусственно дробили промт — оператор хотел переплести
стиль с запретами / разместить пример в середине биографии и т.п. Нельзя.

С одним textarea — пишешь как хочешь, система просто подставит значения
плейсхолдеров. Доверие оператору; превью покажет итог до сохранения.

### Дефолтов нет
По требованию: пустой воркер = пустой промт. Без overlay'я с
"Запретами" и "Форматом" как было в 9.0. Если оператор поленился — гейты
блокируют запуск (понятная ошибка вместо тихой генерации мусора).

### Плейсхолдеры — расширенный набор
Было: только `{current_time}` и `{conversation_history}`.

Стало:
- `{current_time}` — UTC время DD.MM.YYYY HH:MM:SS
- `{worker_name}` — `accounts.name` нашего воркера
- `{partner_username}` — без `@`, или пусто
- `{partner_name}` — `first_name + last_name` (trimmed)
- `{partner_bio}` — Telegram bio
- `{conversation_history}` — формат блоков по `docs/history_format_spec.md`
- `{messages_count}` — `COUNT(*)` по диалогу
- `{days_since_first}` — `(now - MIN(date)).days`

`partner_*` доступны и в initial (берутся из resolved profile),
и в reply (из dialogs row). `conversation_history`/`messages_count`/
`days_since_first` — только в reply (требуют существующего диалога).

### Кнопки "Скопировать" вместо вставки в курсор
По UX-просьбе: оператор сам решит куда вставить, кнопка просто кладёт
`{name}` в буфер. Cleaner и интуитивнее чем магическая вставка.
Подтверждение копирования — иконка-галочка на 1.2с.

### Превью двух блоков, не одного
Раньше показывали один system + user-комментарий "Ответь сейчас.".
Сейчас два независимых блока (initial / reply) — оператор сразу видит
оба итога. user-сообщения постоянны, в превью не показываем (шум).

### `partner_*` для initial — из resolved profile, не из БД
Когда оператор жмёт "+ Новый авто-диалог", диалога ещё нет.
`wrapper.resolve_username(@durov)` возвращает Telegram-профиль —
из него и берём `first_name`, `last_name`, `bio`. После создания диалога
в `dialogs` те же данные осядут — для следующих ответов уже из БД.

### Старые данные стёрты, не мигрированы
По требованию: "можем удалить, не проблема". Миграция 0006 просто
делает DROP COLUMN — никаких склеек 8 полей в один текст. Все 8
воркеров получают пустой промт; нужно заполнить заново.

## Гейты (без изменений)
- Пустой `initial_template` + `/autochat/start` → 409
  `PROMPT_NOT_CONFIGURED`.
- Пустой `reply_template` + входящее сообщение →
  `autochat.generation_skipped` в Event Log, генерация пропускается.

## Что НЕ сделано (намеренно)

- Нет дефолтного шаблона при первом открытии — оператор сам пишет.
- Нет валидаторов "должен содержать <msg>" — превью покажет итог,
  оператор решает.
- Кнопки-вставки плейсхолдеров в позицию курсора — заменены на копирование
  в буфер по запросу оператора.
- Старые данные не мигрировали (по требованию — стерли).
