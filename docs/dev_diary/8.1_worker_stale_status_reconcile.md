# 8.1 — WorkerManager: reconcile stale статусов при старте

**Дата:** 2026-04-24
**Этап:** 8.1
**Коммит:** `83a056c`

## Что сделано

Один метод `WorkerManager.reconcile_on_boot()` и один `await` в lifespan — чинит расхождение между Redis (stale `running`) и in-memory `_slots` (пусто после рестарта контейнера).

## Файлы

- [modules/worker_manager/service.py:104-132](../../modules/worker_manager/service.py#L104-L132) — новый метод.
- [api/main.py:73-77](../../api/main.py#L73-L77) — вызов из lifespan после создания WorkerManager, до того как FastAPI начинает обслуживать запросы.

## Проблема

После перезапуска app-контейнера:

1. `_slots = {}` — пусто в памяти.
2. В Redis ключи `worker:{id}:status` остались с прошлого запуска (TTL нет).
3. `list_workers()` читает статус из Redis → возвращает `status=running, uptime_seconds=0`.
4. UI показывает карточку с зелёной точкой и кнопкой Stop.
5. Пользователь жмёт Stop → POST `/workers/1/stop` → `self._slots.get(1)` → `None` → `raise NotRunning()` → **409**.
6. UI глотает ошибку через `.catch(() => {})` (отдельный UX-баг, разберу в 8.2) → кнопка «не работает».

Живой пример: через `curl` до фикса —
```
GET /workers → [{ "status": "running", "uptime_seconds": 0 }]
POST /workers/1/stop → 409 {"error":{"code":"NOT_RUNNING"}}
```
`uptime_seconds=0` при `status=running` — чёткая метка stale (реально работающий воркер всегда имеет положительный uptime).

## Решение

При старте приложения, ДО того как API начнёт обслуживать запросы, пробегаем по всем аккаунтам и сбрасываем не-терминальные статусы в `stopped`:

```python
async def reconcile_on_boot(self) -> None:
    pool = db.get_pool()
    async with pool.acquire() as conn:
        rows = await conn.fetch("SELECT id FROM accounts")
    changed: list[int] = []
    for r in rows:
        status = await self._get_status(r["id"])
        if status in (STATUS_RUNNING, STATUS_STARTING, STATUS_STOPPING):
            await self._set_status(r["id"], STATUS_STOPPED)
            changed.append(r["id"])
    if changed:
        log.info("reconcile_on_boot: reset stale status for accounts=%s", changed)
```

И в lifespan:
```python
app.state.worker_manager = WorkerManager()
await app.state.worker_manager.reconcile_on_boot()
```

## Почему так, а не иначе

### Почему не авто-рестарт воркеров при старте

Соблазн: если в Redis `running`, поднять воркер самому. Три причины против:

1. **Stacked подпись в UI спеке** — воркеры запускаются вручную через Start, это осознанное решение (см. `docs/worker_manager.md` если добавишь). Авто-старт шёл бы против пользовательского контракта.
2. **Побочные эффекты**: старт воркера дёргает Telegram API, инициирует коннект, может поднять session_expired. Делать это без явного пуша кнопки — сюрприз на каждом `docker compose restart`.
3. **Симметрия с autochat sessions**: в 7.7 мы явно написали что auto-start воркеров не скоуп этапа (см. "Открытые вопросы"). Reconcile «в stopped» — достаточное и честное состояние.

### Почему не TTL на Redis-ключе

Альтернатива — ставить TTL на `worker:{id}:status` так чтобы он истекал быстрее чем typical uptime (например 60 сек), и живой воркер периодически его продлевал. Работает, но:
- Нужен фоновый heartbeat-таск в каждом воркере (ещё одна точка отказа).
- Нагрузка на Redis пропорциональна числу воркеров × частоте heartbeat'а.
- Пока воркеров штуки — reconcile на старте проще и честнее.

Вернёмся к TTL/heartbeat если воркеров станет десятки и шагать по всем аккаунтам в lifespan станет заметно по времени.

### Почему именно `STARTING/STOPPING/RUNNING` → `stopped`

- `RUNNING` — явно stale если slot пустой.
- `STARTING` — transition-state. Если контейнер упал посреди старта, оно так и осталось. Сбрасываем.
- `STOPPING` — аналогично, transition. Сбрасываем.
- `STOPPED`, `CRASHED`, `SESSION_EXPIRED` — терминальные, **не трогаем**. Они отражают реальную историю: CRASHED с `last_error` информативнее чем голое STOPPED. SESSION_EXPIRED — единственный сигнал что требуется reauth.

### Почему не в конструкторе `WorkerManager.__init__`

Конструктор синхронный. DB pool и Redis client к этому моменту уже инициализированы (см. порядок в `lifespan`), но запрос из `__init__` потребовал бы `asyncio.run_coroutine_threadsafe` или `asyncio.get_event_loop().run_until_complete` — некрасиво. Чистый `async def reconcile_on_boot()` вызываемый явно из lifespan проще читать.

## Открытые вопросы

- **Аналогичная проблема у autochat sessions**: уже частично решена в [7.7](7.7_autochat_fixes.md#фикс-2) через `_restore_active_sessions()`. Там не reconcile в stopped, а восстановление в active — другая семантика, потому что сессии переживают рестарт (их состояние в БД, не только в Redis).
- **Интеграционный тест на reconcile**: нужен тест который (а) записывает `worker:1:status = running` в Redis, (б) поднимает приложение, (в) проверяет что статус стал `stopped`. Сейчас проверяли только вручную через curl.
- **UX баг в UI (8.2)**: при 409/5xx от /workers/stop UI молчит. Починить отдельным коммитом.
