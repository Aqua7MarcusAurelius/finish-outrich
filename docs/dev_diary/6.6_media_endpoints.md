# 6.6-6.7 — Endpoints /media/* (GET, file, retranscribe, redescribe)

**Дата:** 2026-04-22
**Этап:** 6

## Что сделано

Четыре endpoint'а группы `/media` в одном новом файле `api/routes/media.py`:

- `GET /media/{id}` — метаданные + вычисляемое поле `file_available`.
- `GET /media/{id}/file` — стрим файла из MinIO с правильным
  `Content-Type` и `Content-Disposition`.
- `POST /media/{id}/retranscribe` — сбрасывает `transcription_status → pending`
  и публикует `media.reprocess.requested [field=transcription]`.
- `POST /media/{id}/redescribe` — то же самое для описания.

Оба *.reprocess endpoint'а объединены в общую функцию `_reprocess(media_id,
field)` — код один, только таблица статуса и set типов разные.

## Файлы

- `api/routes/media.py` — новый роутер.
- `api/main.py` — импорт и регистрация роутера.

## Почему так, а не иначе

### Один файл на всю группу, не по одному на endpoint

Все 4 endpoint'а работают с одной таблицей `media`, три из них делят
валидацию (media существует → тип подходит → файл не удалён). Разбивать
по файлам на этом этапе — лишняя абстракция. Если появится ещё 5-10
endpoint'ов — тогда разобьём.

### Стрим через StreamingResponse с одним чанком

По доке `api.md` — «стрим из MinIO». Реально мы уже читаем объект целиком
в память (`minio_mod.get_object` → bytes). Файлы у Telegram ограничены
20 МБ (регулярные) / 2 ГБ (Premium). Для текущего масштаба лишнее
истинное стриминг-чтение — преждевременная оптимизация.

Возвращаем `StreamingResponse` с генератором на один чанк — так
сохраняется формальный контракт (можно в будущем переключиться на
честный стрим без изменения сигнатуры endpoint'а), клиент читает
через обычный `Content-Length`.

### file_available вычисляется в _media_to_dict

Правило простое: `storage_key is not None AND file_deleted_at is None`.
Дублирует логику из `/media/{id}/file` (там же проверяется), но это
ровно один if в одном месте — не зло.

### _reprocess единый, set типов локальный

В сервисах (transcription/media_description) есть свои наборы поддерживаемых
типов. Можно было бы импортировать. Решил продублировать в
`api/routes/media.py`:

- API валидирует вход **до** похода в сервис — это нормальный слой
  контроля, он имеет право на свои определения.
- Избегаем тонкой зависимости api → modules (api.routes не должен
  трогать внутренности сервисов).
- Наборы меняются очень редко; когда изменятся — поправим в двух местах.

### WRONG_MEDIA_TYPE для не того типа

Например, `retranscribe` на фото → 409 `WRONG_MEDIA_TYPE`. Не 400 и не
404, потому что media существует, просто у неё нет аудиодорожки. 409 —
конфликт состояния, правильно по семантике.

### 410 FILE_CLEANED при отсутствии файла

Если storage_key NULL или файл удалён чистильщиком — 410 с сообщением
«Файл удалён, метаданные и транскрипт остались». По `docs/api.md`.

Применяется и для `/file`, и для *.reprocess endpoint'ов — без файла
обрабатывать нечего.

## Тестирование

Все проверки выполнены из контейнера (через `docker exec tgf_app curl`),
потому что локальный bash/curl на Windows портит UTF-8 в отображении
(cp1251 codepage) — артефакт терминала, не API.

| Проверка | Результат |
|---|---|
| `GET /media/26` | 200, все поля, `file_available: true`, description на кириллице читаемый |
| `GET /media/9999` | 404 `MEDIA_NOT_FOUND` |
| `HEAD /media/26/file` | 405 (endpoint только GET — ожидаемо) |
| `POST /media/26/redescribe` | 200, `{status: "pending", event_id: ...}` |
| Цепочка после redescribe | `media.reprocess.requested → description.started → description.done → message.updated` — всё прошло, media.description перезаписано новым текстом от GPT-4o |

Endpoint `retranscribe` не тестировал точечно — логика симметричная,
слушатель в TranscriptionService уже работает (проверено в Этапе 5).

## Замечание на будущее

`GET /media/{id}/file` — при росте файлов до сотен мегабайт стоит
переписать на честный стрим через `minio.get_object` без `.read()`
(оно возвращает response с `stream(chunk_size)`). Текущая реализация
держит файл целиком в памяти FastAPI-процесса на время передачи
клиенту. Для 20 МБ это нормально, для 2 ГБ Premium-файлов — нет.
