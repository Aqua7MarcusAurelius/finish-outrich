## Что сделано

Ядро работы с LLM: сбор контекста диалога для `chat_completion`,
отдельный путь для первого сообщения, парсер `<msg>`-тегов из ответа.
Плюс иерархия ошибок модуля. Чистые функции без внешних side-effects
(за исключением read-only доступа к БД).

## Ключевые решения

### Сборка контекста — один SQL на messages, один на media

```python
msg_rows = await conn.fetch(
    "SELECT id, is_outgoing, date, text FROM messages WHERE ..."
)
media_rows = await conn.fetch(
    "SELECT * FROM media WHERE message_id = ANY($1::int[])"
)
```

Альтернатива — `JOIN` в одном запросе с `jsonb_agg`, но код читается
хуже, а производительность для `LIMIT 200` одинаковая. Два запроса
проще поддерживать.

### Лимит 200 последних сообщений

Opus 4.7 с контекстом 200k — технически выдержит и тысячу, но:
- payload через base64 раздувает запрос;
- стоимость запроса линейна по токенам;
- полезность падает — для живого диалога обычно важен свежий
  контекст (последние 20-50 сообщений).

200 — с запасом для сложных переписок, с защитой от "захлёба" на
старых диалогах.

### Медиа как текстовые пометки

```
[голос 12с: «привет как дела»]
[фото: «Серый котёнок держит в лапах нож»]
[документ application/pdf: «договор аренды квартиры на 2026 год»]
```

**Почему не передавать реальные картинки/аудио в мультимодальный
запрос:** Opus 4.7 не vision-модель, работает только с текстом.
У нас уже есть транскрипции/описания через `transcription` и
`media_description` модули — их и используем.

Обработка pending/failed:
- `pending` → "[голос: расшифровывается…]" (это временно, planner
  ждёт `message.updated` перед LLM);
- `failed` → "[голос: не удалось распознать]" (LLM видит что было
  что-то, но не смогли разобрать — ответит общими словами).

### Видео → и описание, и транскрипция

```
[видео 30с: «Мужчина едет на велосипеде»; звук: «ребята, смотрите»]
```

Для видео оба поля важны — "что показано" и "что говорят".
У `video_note` (кружок) аналогично.

### `build_initial_messages` — отдельно от `build_conversation_context`

Первое сообщение не должно быть:
- стеной из 4 сегментов незнакомому человеку;
- с `<msg>`-тегами (простой текст);
- с приветствиями-шаблонами ("Здравствуйте, меня зовут Василий!").

Отдельная функция с другой system-инструкцией: "одно живое
короткое сообщение, без markdown, без тегов". В `system_prompt`
пользователя всё равно остаётся — персонаж и правила берутся из
его промта.

### Парсер `<msg>` — regex + fallback

```python
_MSG_TAG_RE = re.compile(r"<msg>(.*?)</msg>", re.DOTALL | re.IGNORECASE)
```

`re.DOTALL` — чтобы `.` матчило перенос строк (у Opus сегменты
могут быть многострочными). `re.IGNORECASE` — на случай `<MSG>`
или `<Msg>`.

Fallback: если тегов не нашли — весь ответ = один сегмент (после
`strip`, с удалением "голых" тегов которые модель всё-таки могла
случайно оставить).

### Safety: `MAX_SEGMENTS = 8`, `MAX_SEGMENT_LEN = 2000`

На случай если модель взбесится и нагенерит 50 сегментов или
сегмент длиной в роман. Не хочется выстрелить в ногу.

### `sanitize_initial_response`

Для первого сообщения: если Opus всё-таки вставил `<msg>`-теги
(в initial промте их нет, но модель может проигнорировать), снимаем
теги и склеиваем через пробел. Одно сообщение, один visual bubble
в Telegram.

## Иерархия ошибок

```
AutoChatError (base)
├── AccountNotFound          404
├── WorkerNotRunning         409
├── SessionAlreadyActive     409
├── UsernameNotFoundError    404
├── UsernameUnavailableError 400
├── CannotWrite              409  ← заблокировали / privacy
├── GenerationFailed         502  ← OpenRouter лёг
├── SessionNotFound          404
└── SessionExpired           410  ← TG-сессия воркера протухла
```

Те же поля (`code` / `message` / `status_code`) что у `AuthError` —
чтобы роутер мог использовать один и тот же маппер в JSON. Паттерн
проверен на `modules/auth/routes.py::_err`.
