# 7.4 — Session state machine + Service orchestrator

**Дата:** 2026-04-22
**Этап:** 7.4
**Коммит:** `c3fb285 feat(autochat): stage 7.4 — session state machine + service`

## Что сделано

Два самых больших файла модуля:

- **`session.py`** (`AutoChatSession`) — per-сессионная state machine
  с тремя параллельными asyncio-контурами внутри одной сессии.
- **`service.py`** (`AutoChatService`) — глобальный оркестратор:
  consumer шины + индексы сессий + публичный API (`create_session`,
  `stop_session`, `list_sessions`, `get_session`).

## Файлы

- `modules/autochat/session.py` — 650+ строк с комментариями.
- `modules/autochat/service.py` — 500+ строк.

## Ключевые решения

### Три контура в одной сессии — зачем

```
state_loop    ── читает events из asyncio.Queue, двигает state,
                 запускает/гасит вспомогательные таймеры (enter/idle),
                 дёргает planner через _planner_bump.set()

planner_loop  ── ждёт bump → ждёт reply_timer (30с тишины) → проверяет
                 pending media → LLM → parse_segments → кладёт в
                 send_queue. Может запускать новую генерацию ПОКА
                 sender ещё отправляет предыдущую.

sender_loop   ── читает сегменты из send_queue. Строго последовательно:
                 set_typing → sleep(typing_duration) → cancel_typing
                 → send_message → publish message.received → sleep(jitter).
```

**Почему разделили planner и sender** (не один cycle: bump → LLM →
send → bump → LLM → send):

Из пункта 3 обсуждения с юзером — "собеседник пишет пока мы
отправляем прошлый ответ, это даже натурально". В объединённой
модели пока работает send — planner не может начать новую генерацию.
С разделением: пришло новое сообщение → сразу запускается новый
reply_timer → через 30с пошла генерация → результат лёг в очередь
→ sender сам до неё доберётся когда закончит текущий set.

**Почему сериализация отправки критична:** без неё при 2 параллельных
генерациях может пойти "второй ответ начал печатать пока первый
ещё не закончил" — в UI Telegram это выглядит как глюк. Через одну
очередь и один sender-loop — получается натуральное "две пачки
ответов подряд", что и хотели.

### Bump-pattern для reply_timer

```python
while True:
    try:
        await asyncio.wait_for(self._planner_bump.wait(), timeout=reply_sec)
    except asyncio.TimeoutError:
        break  # 30с тишины прошли — запускаем генерацию
    self._planner_bump.clear()
    # Был bump — таймер ресетится с нуля
```

Классический паттерн "дебаунс" на `asyncio.Event` + `wait_for` с
таймаутом. События которые дёргают bump:
- новое inbound (`_on_inbound` при `in_chat=True`);
- typing собеседника (`_on_typing`);
- `message.updated` (готовость медиа — возможно переработать ответ).

### _pending_media_msg_ids + _media_ready.Event

Inbound с pending media попадает в `_pending_media_msg_ids`.
Planner после reply_timer проверяет: если set не пустой — ждёт
`_media_ready.Event` (сетится когда последний pending убран в
`_on_media_updated`). Таймаут 2 минуты — чтобы не зависнуть навечно
если транскрипция-модуль умер.

Проверку pending делаем через БД в `_on_media_updated` (запрос
`SELECT 1 FROM media WHERE message_id=$1 AND (..._status = 'pending')`)
— не доверяемся payload'у события, потому что у одного сообщения
может быть несколько медиа.

### enter-timer НЕ перезапускается — один таск на очередь

```python
if self._enter_task is None or self._enter_task.done():
    self._enter_task = asyncio.create_task(self._enter_after(delay))
# Иначе — таймер уже идёт, игнорируем входящее
```

Прямая реализация договорённости из 7.0: повторные inbound пока
enter-timer работает не влияют. После входа они обработаются в
reply_timer вместе с первым.

### idle-timer — "грязный" cancel+spawn

```python
def _reset_idle(self) -> None:
    if self._idle_task and not self._idle_task.done():
        self._idle_task.cancel()
    self._idle_task = asyncio.create_task(self._idle_countdown())
```

Каждое сообщение (наше или собеседника) перезапускает idle-таск
с нуля. `CancelledError` в `_idle_countdown` глушится, `in_chat=False`
ставится только если дождались конца отсчёта.

### Эмиссия `message.received` для собственных отправок — в sender

После `wrapper.send_message(...)` sender публикует `message.received`
с `is_outgoing=True` — точь-в-точь как делает `history/routes.py::send_message`
и `worker.py`. Логика в `history_service` запишет отправку в БД,
и следующий контекст LLM увидит свой же ответ.

**Открытый вопрос из 7.0 закрыт так:** эмитим в AutoChat, не в
wrapper. Причина — в wrapper уже есть предположение, что эмиссия
делается "по требованию" места вызова (`history/routes.py`). Два
похожих фрагмента в разных модулях — ок для MVP; выносить в wrapper
централизованно можно позже, это рефакторинг без изменения поведения.

### state/planner/sender координируются через два asyncio.Event и две asyncio.Queue

```
_queue            (asyncio.Queue)  ← события из service'а → state_loop
_send_queue       (asyncio.Queue)  ← сегменты из planner'а → sender_loop
_planner_bump     (asyncio.Event)  ← триггер от state_loop'а → planner
_media_ready      (asyncio.Event)  ← сигнал от state_loop → planner (media готово)
```

Единый владелец мутабельного state — сам `AutoChatSession`. Остальные
таски читают через эти 4 примитива. `_persist_state()` пишет в БД
под каждое изменение — rollback при рестарте аккуратный.

### Фатальные ошибки — `_fail(reason)`

При `SessionExpired` в send или generate: `status → 'failed'`,
`autochat.session_error` на шину, `asyncio.create_task(self.stop())`
в фоне. Не дожидаемся себя (`await self.stop()` из sender_loop — тупик,
sender_loop сам бы себя ждал).

### AutoChatService — два индекса

```python
_sessions_by_id:  {session_id: AutoChatSession}
_by_dialog:       {(account_id, dialog_id):       session_id}
_by_tg_user:      {(account_id, telegram_user_id): session_id}
```

Зачем **оба**:
- `message.saved` / `message.updated` приходят с `dialog_id`, но после
  первой отправки dialog_id ещё не известен (history не успел
  создать dialog). Фолбэк — по `telegram_user_id`, потом привязываем
  `dialog_id`.
- `dialog.typing_observed` приходит с `telegram_user_id` (wrapper
  не знает dialog_id).

Индексы защищены `asyncio.Lock` — на всякий случай для будущих правок,
хотя сейчас все мутации идут из consumer_loop сервиса (один таск).

### `create_session` — полный синхронный флоу

```
1. проверка worker_running      (быстро)
2. resolve_username             (1 сетевой)
3. SELECT дубликата             (быстро)
4. build_initial_messages + LLM (5–30 сек)
5. INSERT ... ON CONFLICT ... WHERE status IN (...)
6. wrapper.send_message         (1 сетевой)
7. publish autochat.started
8. publish message.received     (history запишет)
9. publish autochat.initial_sent
10. UPDATE status='active'
11. создать AutoChatSession, зарегистрировать в индексах, start()
```

Клиент ждёт весь флоу — но это естественно: `/autochat/start`
возвращает готовую сессию, а не "будет готово в фоне". 5-30 секунд
приемлемо для старта переписки.

**Защита от гонки** — partial unique index по `(account_id,
telegram_user_id) WHERE status IN ('starting','active','paused')` +
`INSERT ... ON CONFLICT ... DO NOTHING RETURNING *`. Если одновременно
два запроса с одинаковой парой — первый победит, второй получит
`None` из `fetchrow` и вернёт `SessionAlreadyActive`.

Почему `status='starting'` сразу после INSERT, а не `'active'`: между
INSERT и send может улететь исключение (сессия TG протухла, лётбан
за флуд). Если сразу `'active'` — при падении send останется
висячий 'active' без запущенного таска. `'starting'` → `'active'`
только после успешного send: инвариант "status='active' ⇒ есть
задача" сохраняется.

### Восстановление после рестарта — отложено

В lifespan при старте `AutoChatService` не загружает активные сессии
из БД. Пока — на втором заходе. Для MVP с 1–2 сессиями проще
руками пересоздать после рестарта, чем заложить сложный restore
(пересчёт in_chat, возраста сообщений, отвалившихся таймеров).

Заложил комментарий в service.py → run(): _restore_on_start() хук
для будущего.

## Открытые вопросы

- **Дебаунс typing-bump.** Телеграм шлёт typing-события каждые ~5 сек
  пока собеседник печатает. Это нормально тригерит bump (ресет
  reply_timer). Но если собеседник пишет длинное сообщение 2 минуты —
  reply_timer будет ресетиться 24 раза. На поведение не влияет
  (таймер и так должен ждать), но события `dialog.typing_observed` и
  последующие bump'ы в planner'е сильно засорят архив. Если
  окажется проблемой — добавлю дебаунс 2 сек на уровне handler'а
  wrapper'а.

- **Восстановление после рестарта.** Сейчас завалившиеся с
  status='active' сессии остаются в БД. В MVP — удобно для аудита
  ("что было на момент падения"). Перед мержем в main — решу:
  автоматический restore или CLI-команда "restart all active
  sessions after outage".
