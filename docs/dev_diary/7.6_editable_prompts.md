# 7.6 — Редактируемые промты в текстовых файлах

**Дата:** 2026-04-22
**Этап:** 7.6 (перед живым прогоном)
**Коммит:** будет после этой записи

## Что сделано

Промты AutoChat вынесены из кода в `prompts/*.md`-файлы. Пользователь
может редактировать их любым текстовым редактором — следующая генерация
ответа подхватит новый текст без рестарта приложения.

Дополнительно: поддержка опционального плейсхолдера
`{conversation_history}` — если он присутствует в reply-промте, история
переписки подставляется в system-промт текстом, а role-messages не
используются. Убрал плейсхолдер — стандартный режим с ролями.

## Файлы

- `prompts/autochat_reply_system.md` (новый) — для ответов.
- `prompts/autochat_initial_system.md` (новый) — для первого сообщения.
- `modules/autochat/generation.py` — полностью переписан: чтение файлов,
  render плейсхолдеров, два режима истории.
- `docs/autochat.md` — новый раздел "Редактирование промтов" с таблицей
  плейсхолдеров; старый раздел "Промт сегментации" стал ссылкой на файл.

## Почему именно так

### Чтение при каждой генерации, не кэш

Overhead: файл 1–2 КБ, `Path.read_text()` ~0.1 мс. Для сессии с 1–2
диалогами, где генерация раз в минуту — это копейки. Плюсы:
- отредактировал → сохранил → следующая сессия с новым текстом;
- не надо дёргать никаких `reload` endpoint'ов;
- если файл временно побит (незакрытая скобка плейсхолдера) —
  только текущий запрос получит странный текст, следующий после
  исправления будет ок.

Альтернатива — кэш с mtime-check. Оставил на потом, если будет
реально узкое место. Сейчас не нужно.

### Плейсхолдеры `{key}`, не Jinja

Хотел максимально простой формат. Jinja/Mako — это:
- +зависимость;
- +синтаксические ошибки в шаблоне, которые пользователь не заметит;
- +логика в шаблоне (условия, циклы) которая нам сейчас не нужна.

`re.sub(r'\{(\w+)\}', ...)` с fallback "неизвестный ключ оставляем как
есть" — 10 строк кода, пользователь видит `{unknown}` в ответе и сразу
понимает что опечатался в имени плейсхолдера.

### HTML-комментарии `<!-- ... -->` вырезаются

В файлах промтов есть инструкции для редактора — "опиши персонажа
здесь", "какие есть плейсхолдеры", "формат" и т.п. Если отправить их
в LLM как есть — модель будет на них реагировать как на часть промта.

Решение: перед отправкой в LLM `_strip_comments()` убирает всё между
`<!--` и `-->`. Пользователь может писать мета-заметки прямо в файле,
они не попадают в запрос.

### Два режима истории — через наличие `{conversation_history}`

Пользователь спросил "как лучше". Честно: **role-based обычно лучше**
(LLM разработана под этот формат, метка "assistant/user" даёт чёткое
разделение). Но он хочет гибкости — иногда удобнее написать промт в
стиле "Вот история: ... Теперь ты ... ответь".

Компромисс: наличие плейсхолдера в файле выбирает режим.

- `{conversation_history}` есть → собираем всю переписку текстом
  "(дата) Ты/Собеседник: ...", подставляем в system; messages = [system,
  user:"Ответь сейчас."]. Всё, что хочешь — в system.
- Плейсхолдера нет → стандарт с ролями.

Одна функция `_has_placeholder(template, key)`, одна проверка, две
ветки.

### Fallback при отсутствии файла

Если пользователь случайно удалит `prompts/autochat_reply_system.md`
или накосячит с правами на файл — система не падает. Логируется
warning `autochat: prompt file not found (...), using fallback`, и
используется минимальный встроенный промт (`_REPLY_FALLBACK` в
`generation.py`) — с сегментацией и простейшими правилами.

Ответы будут скучными, но модуль работает. Это лучше чем "вся
автообщение застряло потому что файл куда-то делся".

### Что остаётся в коде

- `_REPLY_FALLBACK` / `_INITIAL_FALLBACK` — минимальный промт на случай
  отсутствия файла.
- Формат `{current_time}` — `"%Y-%m-%d %H:%M"` UTC.
- Формат истории в тексте — `"(YYYY-MM-DD HH:MM) Ты/Собеседник: ..."`.
- Медиа как текст — "[голос 12с: «расшифровка»]" и т.д. (оставлено
  как было в 7.3).

Это "технические решения" — не думаю что пользователь хочет их
редактировать. Если захочет — видно сразу в `generation.py`.

## Интеграция с существующим 7.4

`session.py::_generate_and_enqueue()` вызывает `build_conversation_context`.
Функция унаследовала публичный контракт — возвращает `list[{role, content}]`
как и раньше. Режим A (история в system) = ещё один валидный
вариант этого списка (один user-message). Планнеру всё равно, LLM всё
равно — меняется только содержимое.

Никаких правок в `session.py` / `service.py` / `routes.py` не
потребовалось.

## Docker

`Dockerfile` содержит `COPY . .` — `prompts/` попадает в образ при
билде. `docker-compose.yml` монтирует `./:/app` — папка доступна и
через volume. `.dockerignore` не содержит `prompts` — всё ок.

На рестарт контейнера не нужно заново билдить образ: файлы промтов
видны через volume.

## Smoke-тесты (без контейнера)

Прогнал 7 проверок:
1. `_strip_comments` — HTML-комментарии удаляются корректно.
2. `_render` — подстановка работает, отсутствующий ключ сохраняется
   как `{unknown}`.
3. `_has_placeholder` — детектит присутствие `{key}`.
4. `_format_history_text` — форматирует turns в
   "(дата) Ты/Собеседник: ...", пустой список → "(история пустая...)".
5. Чтение `prompts/autochat_reply_system.md` — 1465 байт после
   strip-комментариев, `<!--` не осталось, плейсхолдеры на месте.
6. Чтение `prompts/autochat_initial_system.md` — 712 байт, ок.
7. Fallback при отсутствии файла — возвращает встроенный текст.

Живой прогон через Opus (7.7) — после рестарта контейнера
пользователем.
