# 6.1 — OpenRouter: функции описания изображений и документов

**Дата:** 2026-04-22
**Этап:** 6 (модуль описания медиа)

## Что сделано

Расширен клиент OpenRouter двумя функциями для модуля описания:
`describe_images` (фото/стикеры/кадры) и `describe_document` (PDF и др.
документы через Gemini с нативной поддержкой файлов). Общие части клиента
(заголовки, POST, парсинг ответа) вынесены в приватные хелперы — чтобы
не дублировать их теперь трижды.

## Файлы

- `core/openrouter.py` — добавлены `describe_images`, `describe_document`,
  приватные `_headers`, `_post`; промпты `IMAGE_DESCRIPTION_PROMPT` и
  `DOCUMENT_DESCRIPTION_PROMPT`. `transcribe_audio` переведена на общий
  `_post` (поведение не изменилось).
- `core/config.py` — новая переменная `OPENROUTER_MODEL_DESCRIPTION_DOCUMENTS`
  (по умолчанию `google/gemini-2.5-flash`).
- `.env.example` — та же переменная с комментарием.

## Почему так, а не иначе

### Две модели вместо одной

`openai/gpt-4o` в OpenRouter отлично принимает картинки через `image_url` с
data-URL, но **не** принимает документы напрямую — для PDF нужна модель с
нативной поддержкой файлов. Из доступных в OpenRouter — Gemini (`google/
gemini-2.5-flash`) и Claude. Выбран Gemini: дешевле, быстрее, и по
документации OpenRouter PDF проходит нативно без плагина.

Альтернатива «одна модель для всего» (только gpt-4o + пропускать
документы) была отклонена — документы в Telegram всё-таки бывают,
терять их целиком некрасиво.

### plugins.file-parser включён

В `describe_document` добавлен `plugins: [{id: "file-parser", pdf: {engine:
"native"}}]`. Для Gemini это no-op (файл и так идёт нативно), но если
в будущем модель поменяется на ту, что PDF не умеет, — OpenRouter
автоматически подключит парсер. Страховка без оверхеда.

### Промпты на русском

Выход модели попадает в поле `media.description`, которое потом
отображается в UI на русском. Промпт на русском гарантирует русский
ответ без лишних «I see in the image...».

### Общий `_post`

Транскрипция и два новых метода все шлют в один и тот же endpoint с
одинаковыми заголовками и одинаковым форматом ответа. Хелпер `_post`
избавляет от трёх копий одной и той же обработки ошибок и парсинга
`choices[0].message.content`. Нарушений «не рефакторить сверх задачи»
нет — без хелпера 70 строк были бы скопированы.

## Открытые вопросы

- **Не-PDF документы (xlsx, docx).** Gemini по документации умеет часть
  форматов нативно, но не все. Точный список не зафиксирован — узнаем
  на живом тесте (см. план для 6.5). Если не поедет — будет
  `description.done [failed]` с понятным error-текстом, система при
  этом работает.
- **Размер документа.** Лимит не поставлен (по доке `media_description.md`
  «размер не ограничиваем»). У Gemini контекст большой, но payload в
  base64 через HTTP — не миллиарды байт. Если встретятся 100+ МБ PDF —
  вернёмся.
