# 7.8 — Живой тест: копирование timestamp'ов и 400 на ручных outgoing

**Дата:** 2026-04-24
**Этап:** 7.8
**Коммит:** (pending)

## Что произошло

Живой тест сессии `@vians777` (id=2, создана через `POST /autochat/start`)
вскрыл две проблемы. Одну починили, вторую локализовали — фикс в
следующем заходе.

## Проблема 1 — Opus копирует timestamp-префиксы из истории в свой ответ

### Как воспроизвелось

После коммита 7.7 каждое сообщение в role-based истории приходило в
LLM с префиксом `[2026-04-24 10:13] тело сообщения`. Opus на ответе с
`segments_count=1` скопировал префикс **внутрь** `<msg>`-сегмента.
Наружу в Telegram ушло:

```
[2026-04-24 10:13] оо круто!
[2026-04-24 10:13] а где катаешь? бали? португалия?
[2026-04-24 10:13] я сама спортивная, но до сёрфа руки не дошли пока
```

Во второй генерации такое же — `[2026-04-24 10:16] вау, целый год!
[2026-04-16 10:16] а не скучаешь ...` (даже дата галлюцинирована).

### Корень

Префикс `[YYYY-MM-DD HH:MM]` **в начале** содержимого Opus принял за
часть формата сообщения — не как служебные метаданные. Особенно
срабатывало на коротких ответах, где меньше контекста указывает на
«правильный» формат.

### Фикс

Два изменения, работают вместе:

**1.** [modules/autochat/generation.py:302-312](../../modules/autochat/generation.py#L302-L312)
— timestamp переехал **в конец** содержимого с прозовым маркером:

```python
content = f"{t['body']}\n(отправлено {stamp})"
```

Было `[2026-04-24 10:13] текст` → стало `текст\n(отправлено 2026-04-24 10:13)`.
Слово «отправлено» даёт Opus'у явный сигнал «это метаданные»,
позиция в конце — снижает вероятность восприятия как часть формата.

**2.** [prompts/autochat_reply_system.md](../../prompts/autochat_reply_system.md)
— добавлен явный пункт в раздел «Формат ответа»:

> НЕ копируй в свои ответы служебные метки времени. В конце каждого
> сообщения истории есть строка вида `(отправлено 2026-04-24 10:13)` —
> это для тебя, чтобы ты понимала когда было написано сообщение. Это
> НЕ часть текста и НЕ часть формата ответа.

Файл читается при каждой генерации (см. 7.6) → фикс включился
мгновенно без рестарта. Последующие 6+ генераций пошли чисто.

### Два сообщения-жертвы

Нельзя откатить — message id=113 и id=122 в диалоге 95 ушли в
Telegram с кривым форматом. В БД они остаются как есть (как факт
отправки); в контексте будущих генераций Opus видит их тоже с
`(отправлено …)` суффиксом, и инструкция не копировать применяется
к ним так же, как ко всему остальному.

## Проблема 2 — HTTP 400 от Anthropic при ручных outgoing

### Как воспроизвелось

Во время живого теста оператор (я — пользователь) **зашёл в аккаунт
ассистента** (`+447350135778`) и руками отправил две GIF-ки через
встроенный `@gif`-inline. События:

- 10:28:51 `message.received` is_outgoing=true, media: gif
- 10:28:57 description.done (готова)
- 10:29:20 autochat.generation_requested (messages_count=62)
- 10:29:31 autochat.session_error — HTTP 400 от OpenRouter/Anthropic:
  `"This model does not support assistant message prefill. The
  conversation must end with a user message."`

Второй GIF — та же история двумя минутами позже.

### Корень

В этой точке истории последнее сообщение в role-based контексте было
`role=assistant` (наша же outgoing-гифка, описанная как
`[gif: «Губка Боб в розовых очках...»]`). Anthropic требует чтобы
последнее сообщение в `messages` было `role=user` (либо поддержка
prefill assistant — чего у этого провайдера нет).

Planner не проверяет кто писал последним перед генерацией. Любой
bump (inbound, typing, message.updated — последнее теперь no-op)
запускает reply_timer. Если за эти 30с нового inbound не пришло —
идём в LLM, с какой бы ролью ни был последний turn.

### Что произошло дальше (анти-паттерн диагностики)

Я ошибочно предположил, что Telethon криво ставит `m.out=True` для
GIF-сообщений от собеседника. Исправил [wrapper.serialize_message](../../modules/worker/wrapper.py#L135)
на cross-check `sender_id == own_user_id`, добавил cache
`own_user_id()` в `TelegramWrapper`, обновил оба хендлера
(`worker.py`, `history_sync/service.py`). Плюс сделал
`UPDATE messages SET is_outgoing=false WHERE id IN (156, 158)` —
«починив» «неправильные» флаги.

Пользователь уточнил: GIF-ки слал **он** из аккаунта Валерии, и
`is_outgoing=true` было абсолютно корректным. 400 — настоящий, но
его причина в другом.

### Откат

- `git checkout -- modules/worker/wrapper.py modules/worker/worker.py
  modules/history_sync/service.py` — вернул три файла к состоянию до
  моих изменений.
- `UPDATE messages SET is_outgoing=true WHERE id IN (156, 158)` —
  вернул флаг.
- Оставил полезные изменения: формат timestamp'а в generation.py и
  инструкцию в промте.

### Настоящий фикс (отложен)

Перед `_generate_and_enqueue` — чтение последнего non-deleted
сообщения диалога. Если `is_outgoing=true` — не генерировать,
вернуться в planner-loop ждать следующего bump'а. Иначе идём в LLM.

Это 3–5 строк в [session.py](../../modules/autochat/session.py) + один
SQL. Нужны также граничные случаи: пустой диалог (ok, generate
начальное), несколько outgoing подряд (ok, не генерим до первого
inbound).

## Выводы для себя

1. **Не диагностировать с налёта.** Первый взгляд на is_outgoing=true
   в payload GIF'а показался явной аномалией Telethon — на самом деле
   я просто не знал что пользователь руками зашёл в аккаунт. 10 минут
   кода, правка БД и лишняя сложность в двух модулях на ровном месте.
2. **Сначала уточнять сценарий, потом фиксить.** Одно сообщение
   оператору — «ты точно не писал от лица ассистента?» — сэкономило
   бы час.
3. **Осторожно с UPDATE в шариковых таблицах.** Я исправил две строки
   `messages` «превентивно». Это обратимо, но в боевых данных
   подобные правки должны делаться только после подтверждения
   гипотезы.
