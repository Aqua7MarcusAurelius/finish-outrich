## Что сделано

Правая колонка страницы «Диалоги» (пустой placeholder из 8.12) заполнена виджетом: архив + live-поток событий шины, отфильтрованных по `data.dialog_id == выбранный_диалог`. Сразу видно что происходит с сообщениями, транскрипциями, описаниями, автодиалогом по конкретному чату — без переключения на страницу Event log.

## Как это работает

### Два канала данных

1. **Архив** через `GET /events?dialog_id=N&limit=50`. `useQuery` с `refetchInterval: 10s` — если live почему-то пропустил событие (реконнект SSE, gap), архив подтянет.
2. **Live** через `GET /events/stream?dialog_id=N`. `EventSource` открывается в `useEffect`, переоткрывается при смене диалога. Буфер 100 событий.

### Merge стратегия

```ts
const map = new Map<string, BusEvent>();
for (const e of live) map.set(e.id, e);         // live приоритетнее
for (const e of archive) if (!map.has(e.id)) map.set(e.id, e);
return [...map.values()].sort((a, b) => b.time.localeCompare(a.time));
```

Дедуп по id. При открытии диалога видишь архив; при появлении новых — они выталкиваются вверх и мигают (animate-flash-new).

### Почему backend-фильтр, а не клиентский

До 8.13 можно было только `?account=N` и фильтровать `data.dialog_id == N` клиентом. На аккаунте с 100+ активными диалогами этот поток был бы 99% мусора. Серверный фильтр (JSONB `(data->>'dialog_id')::bigint = N`) делает стрим узкоспециализированным.

### Почему `shortId` и full id в tooltip

Хеш-UUID 32 символа — не влезает в колонку ~300px. Показываю `#abcdef12`, полный id в `title`. Клик по строке пока не открывает детальный попап (отличается от страницы Event log по скоупу) — но можно добавить, если покажется полезным.

### Empty state

- `dialogId == null` — «Выбери диалог…».
- Diального dialogId нет событий — «событий по этому диалогу пока нет».
- Ошибка архива — `ErrorBox` (мини-версия). Live при этом может продолжать качать.

### refetchInterval 10s — не 5s

10 сек потому что live-поток покрывает 99% случаев. Архивный fetch — страховка от gap'ов, чаще не нужен, а каждые 5 сек шумит по сети. Для страницы Event log использую 15s с тем же мотивом.

## Структура карточки события

```
●  19:14:32  [history]
   message.saved
   сообщение #185 записано в БД
   #e092b643
```

- Status dot — зелёная/красная/жёлтая.
- Время HH:MM:SS.
- ModulePill — цвет модуля шины.
- Type — `event.type` моноширинным жирным.
- Message — human-readable описание (из `core/event_messages.py` через `format_message`).
- Short id моноширинным серым.

Ошибочные события (`status=error`) — вся строка `bg-status-error/10`, чтобы глазом выхватить при быстром скролле.

## Проверка

1. Открыть http://localhost:3000/#/dialogs/1/45 (или любой другой диалог с активностью).
2. Справа от чата должна появиться колонка «Events по диалогу · dialog #45 · live».
3. Архивные события подтягиваются сразу. Если сейчас приходит новое сообщение в этот диалог (и собеседник живой) — увидишь как `message.received` → `message.saved` → возможно `transcription.done` → `autochat.*` всплывают один за другим с миганием.

## Открытые вопросы

- **Клик по событию → детальный попап с цепочкой parent_id.** Можно переиспользовать `EventDetailDialog` с страницы Event log. Не подключал пока, чтобы не раздувать виджет.
- **Типы фильтра.** Сейчас виджет показывает все события по dialog_id. Может захотеться подсечь `module = 'autochat'` или `type = 'transcription.*'`, если один канал заполоняет ленту. Добавить chip-фильтры сверху виджета.
- **SSE-дисконнект.** Сейчас молча закрывается, UI ничего не показывает. Можно добавить индикатор live (зелёная точка) / re-connecting (жёлтая).
- **Clickable module/type для перехода в полный Event log.** «Открыть в Event log этот тип по этому аккаунту» — одна кнопка сверху виджета, которая ведёт на `/events` с нужными фильтрами.
- **freshIds.current рост без границы.** Set растёт пока открыт диалог, очищается только при смене. При длинном сидении с одной открытой лентой потенциально накопит тысячи id. Практически — не страшно, каждый id ~36 байт, тысяча = 36kb. Если станет виднее — очищать при достижении LIVE_BUFFER.
