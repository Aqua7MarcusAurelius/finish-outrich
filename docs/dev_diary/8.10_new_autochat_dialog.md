# 8.10 — «+» под списком диалогов: запуск авто-диалога

**Дата:** 2026-04-24
**Этап:** 8.10
**Коммит:** `0a8fb9e`

## Что сделано

Пользователь может запустить новый авто-диалог прямо из UI — кнопка "+" под списком диалогов в левой колонке открывает модалку, которая дёргает `POST /autochat/start`. Бэкенд делает всё остальное: резолвит @username, генерирует первое сообщение через Opus 4.7, отправляет его, дальше ведёт переписку сам по state machine AutoChatSession.

## Маппинг UI → backend

| Поле UI | В pydantic `StartIn` | Примечание |
|---|---|---|
| Account (выбранный в шапке) | `account_id` | Без выбора кнопка disabled. |
| @username (с @ или без) | `username` | UI снимает ведущий `@`; backend ожидает чистое имя. |
| Initial prompt (обязательно) | `initial_prompt` | min 1 / max 10000 на стороне pydantic. |
| System prompt (опционально) | `system_prompt` | Если пусто — передаём `""`, backend подставит в `{user_system_prompt}` пустую строку, персонаж возьмётся из `prompts/autochat_reply_system.md`. |

## Почему так

### Кнопка под списком, не в шапке рядом с "+" для аккаунта

Для добавления аккаунта кнопка уже в шапке («+ Новый аккаунт» из 8.7). Для нового **диалога** естественный контекст — список диалогов. Плюс кнопка требует выбранного аккаунта — логично чтобы она жила в колонке, которая зависит от account-selection'а. Если вынести в шапку, пользователь будет теряться почему она дизейблится когда нет активного выбора.

### Dashed border

Отличает action от реальных диалогов (которые были бы визуально похожи: аватар + текст). Плюс даёт сигнал «это кнопка создания», а не сущность.

### disabled когда `accountId == null`

Backend валидирует `account_id` через FK на `accounts`. Отправлять без выбранного аккаунта гарантированно 422 — проще предотвратить клик. Tooltip «Выбери аккаунт сверху» заменяет ошибку.

### Не блокируем UI на время резолва @username

`POST /autochat/start` — long-running: resolve_username (~300мс), generate через OpenRouter (~1-5сек), отправка. Модалка показывает `busy` state («отправляем…»), кнопка disabled, но остальной UI живой. Если пользователь закроет модалку — кнопка cancel не умеет прервать уже ушедший POST (в backend сейчас нет `/autochat/{id}/cancel-start` ручки), но результат прилетит в toast когда придёт.

### Два рефетча после success

После `POST /autochat/start` backend создаёт **сессию** мгновенно, но **диалог** (row в `history.dialogs`) появляется только после первого `message.saved` — т.е. после отправки первого сообщения (~1-3 сек). Рефетчу:
1. `accountsQ.refetch()` сразу — чтобы обновился `dialogs_count` и `last_event_at`.
2. `dialogsQ.refetch()` через 2.5с — чтобы новый диалог подтянулся в сайдбар без нажатия F5.

2.5 сек — компромисс: OpenRouter изредка лагает, но на 5с-паузе юзер теряет ощущение «сделалось».

### textarea вручную, не `<Textarea>` shadcn

Не создавал отдельный `textarea.tsx` примитив ради двух использований. Стили скопированы с `Input`, классы одни и те же — если понадобится переиспользовать, выделим.

### Clean username `replace(/^@/, "")`

Пользователь запросто скопирует `@durov` с ведущим собачкой. Backend ждёт `durov`. UI чистит, чтобы не ловить `USERNAME_NOT_FOUND` на ровном месте.

## Проверка

1. Открыть http://localhost:3000/#/dialogs.
2. Убедиться что выбран аккаунт (Test account, зелёная рамка). Иначе "+ Новый авто-диалог" внизу колонки будет серым с tooltip'ом.
3. Клик → модалка. Заполнить:
   - `@username` — любой живой Telegram username (тестовый человек, желательно не бот).
   - `Initial prompt` — например «Напиши дружелюбное первое сообщение, спроси как дела».
   - `System prompt` — пусто (возьмётся Валерия из prompts/autochat_reply_system.md).
4. «Запустить». В течение ~5сек должен прилететь зелёный тост с `session_id`.
5. В течение ~3-5сек новый диалог появится в левой колонке — клик открывает ленту с первым автосгенерированным сообщением.

## Открытые вопросы

- **Нет прогресса во время `POST /autochat/start`.** Сейчас кнопка показывает `отправляем…`, но пользователь не видит что именно идёт — resolve, LLM-generation, или send. Backend для этого не шлёт events обратно по одному HTTP, только общий ответ. Было бы чище подписаться на `/events/stream` с фильтром `account_id` и `autochat.*` в модалке — и показывать внутри неё микрожурнал шагов в реальном времени. Отдельным этапом.
- **Список активных сессий.** UI пока никак не показывает что на аккаунте работают автодиалоги — их надо смотреть через `GET /autochat/sessions?account_id=1`. Было бы ценно: под каждым диалогом (или в заголовке диалога) метка «AutoChat · active», clickable → открывает sidebar с `system_prompt`, `initial_prompt`, `status`, кнопкой Stop. Пригодится для отладки поведения LLM.
- **Остановка авто-диалога.** `POST /autochat/sessions/{id}/stop` есть, кнопки в UI пока нет. Если сессия пошла не туда — сейчас только через curl.
- **Выбор промпта-персонажа.** Системный промпт в UI — голый textarea. Было бы удобно иметь выпадашку с шаблонами из `prompts/autochat_*_system.md` (сейчас там Валерия + дефолт). Для этого нужен endpoint на backend'е который отдаёт список доступных промптов — не делал.
- **Валидация @username.** Можно сразу дёргать `GET /workers/{acc}/resolve/{username}` (если такая ручка появится) чтобы проверить до кнопки «Запустить». Сейчас ошибка прилетит только на submit.
