## Fix 1: `--reload` в Dockerfile

**Симптом:** в логах продакшен-контейнера виднеется `WatchFiles
detected changes ... Reloading`. При любой правке файла uvicorn
перезапускает процесс.

**Где:** `Dockerfile:23`

```dockerfile
CMD ["sh", "-c", "alembic upgrade head && uvicorn ... --port 8000 --reload"]
```

**Почему так вышло:** в `docker-compose.yml:17-18` код монтируется
volume'ом `./:/app`, и с `--reload` получается удобный dev-режим —
правишь файл на хосте, контейнер сам подхватывает. Комментарий в
Dockerfile (L22) говорит «для prod убрать», но это не сделали.

**Почему плохо на проде:**

- Любое сохранение файла рвёт открытые SSE-соединения дашборда.
- Обрывает Telethon-клиентов — сессии с Telegram ломаются.
- Consumer-loop'ы Redis Streams теряют прогресс.
- Watcher файлов жрёт CPU впустую.

**Фикс:** убрали `--reload`. Если нужен hot-reload — запускать
uvicorn локально вне Docker или поднимать `docker-compose.dev.yml` с
override команды. Для одного разработчика первое проще.

После пересборки образа (`docker compose build app`) в логах больше
нет `Started reloader process` — один процесс вместо двух.

Коммит: `6d98033 fix(docker): drop --reload from production CMD`.

## Fix 2: dead-letter для consumer-групп

**Симптом (потенциальный):** четыре consumer loop'а (`history`,
`transcription`, `description`, `archive_writer_loop`) использовали один
и тот же паттерн:

```python
for stream_id, event in batch:
    try:
        await self._handle(event)
        ack_ids.append(stream_id)
    except Exception:
        log.exception(...)
        # Не ack'аем — переобработка в следующем прогоне.
```

Защищает от транзиентных ошибок (БД подвисла, OpenRouter 5xx). Но
если событие **всегда** падает (битый payload, баг в обработчике) —
оно копится в pending-list consumer group вечно. При рестарте с
реплеем pending loop крутится на ядовитом событии. Свежие события по
этому типу тихо пропадают.

Баг в `transcription.done` — и в истории диалога дырки, замечаешь
через месяц.

**Фикс:** добавили в `core/bus.py` два хелпера:

```python
DEAD_LETTER_MAX_RETRIES = 5

async def record_failure(group, stream_id, event, error) -> bool:
    # Инкремент счётчика в Redis (ключ bus:retries:{group}:{stream_id})
    # После N попыток: публикуем system.error с payload
    # {message: "poison_message", group, stream_id, event_id, event_type,
    #  retries, error}, удаляем счётчик, возвращаем True — caller force-ack.

async def record_success(group, stream_id) -> None:
    # Сбрасывает счётчик при успешной обработке.
```

Счётчик в Redis, ключ `bus:retries:{group}:{stream_id}`, TTL 24 часа.
При достижении порога событие принудительно ack'ается, чтобы очередь
не стопорилась, а factum инцидента отражается в `events_archive` как
`system.error`.

Применено во всех четырёх consumer'ах. Изменение точечное — внешние
try/except не трогали, только per-event branch.

### Почему счётчик в Redis, а не в самом событии

- Redis и так под рукой у всех consumer'ов.
- Не надо переделывать формат событий — тот же `publish()`, та же
  сериализация.
- TTL снимает мусор автоматически, если поток замер.

### Почему не dead-letter stream

- Один stream, один источник правды — проще искать события по ID.
- `system.error` с payload уже достаточно для разбора (event_id,
  retries, обрезанный текст ошибки).
- Если захочется реально отдельный поток — вынести потом, интерфейс
  обёртки не изменится.

Обновили `docs/event_bus.md` → добавили секцию про poison messages и
пример строки лога.

Коммит: `7960e81 fix(bus): dead-letter for consumer groups to stop
poison-message blocking`.

## Чистка истории (Вариант B)

Потребовалось перед следующим этапом — чистый слепок с сохранёнными
аккаунтами, без старых сообщений и архива событий.

**Предварительно:** `docker compose stop app`, чтобы воркеры не
писали в БД/MinIO во время чистки.

**Что почистили:**

| Где | Было | Стало |
|---|---|---|
| `accounts` | 1 | **1** (сохранён) |
| `settings` | 7 | **7** (сохранены) |
| `dialogs` | 2 | **0** |
| `messages` | 44 | **0** (каскад от dialogs) |
| `media` | 34 | **0** (каскад от messages) |
| `events_archive` | 343 | **0** (TRUNCATE) |
| Redis keys | 2 | **0** → потом 1 (свежий `events:stream` от нового старта) |
| MinIO objects | 34 | **0** |

Последовательность команд:

1. `docker compose stop app`.
2. MinIO: `mc alias set local ...` → `mc rm --recursive --force
   local/tgframework/`.
3. PostgreSQL: `BEGIN; DELETE FROM dialogs; TRUNCATE events_archive;
   COMMIT;` (одна транзакция).
4. Redis: `DEL events:stream worker:1:status` (остальные префиксы
   `bus:retries:*`, `auth_session:*` оказались пусты).
5. `docker compose up -d app` — новый образ без `--reload`.

**Healthcheck после рестарта:**

```json
{"status":"ok","components":{"postgres":"ok","redis":"ok","minio":"ok","auth_module":"ok"}}
```

Consumer-группы (`archive-writer`, `history-writer`,
`transcription-worker`, `description-worker`) при старте app'а
пересозданы автоматически через `ensure_group()`, все с `pending=0`
и `lag=0`.

### Почему Вариант B, а не полный down -v

- Авторизация Telegram-аккаунтов — долгий ручной процесс через `/auth/*`
  (код, 2FA, два прокси). Если можно сохранить — надо сохранить.
- Сессии Telethon лежат в `accounts.session_data` (bytes). Сам
  `DELETE FROM accounts` каскадно снёс бы всё, включая session_data.
- Настройки `settings` (TTL, retries и т.д.) тоже держатся в БД —
  переливать из миграции и ждать, чтобы не сломать дефолты, лишнее.

Из `docs/AUDIT_REPORT.md`, в порядке приоритета:

   Сейчас клиент создаётся на каждый запрос, нет обработки 429/5xx.

5. Non-root пользователь в Dockerfile.
6. Streaming при `GET /media/{id}/file` — сейчас файл читается в память
   целиком.
7. MinIO bucket policy (deny anonymous) + не выставлять порт 9000 наружу.

Остальное (event versioning, FTS-индекс, валидация `change_me` в
