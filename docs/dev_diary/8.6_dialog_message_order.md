# 8.6 — Диалоги: порядок сообщений как в мессенджере (старые сверху, новые снизу)

**Дата:** 2026-04-24
**Этап:** 8.6
**Коммит:** `50b069a`

## Что сделано

Перевернул визуальный порядок ленты через `flex-col-reverse` — в DOM массив остаётся «новые сверху» (как пришёл от backend'а), но flex-контейнер рендерит их снизу. Плюс скролл сразу оказывается внизу без всяких `useEffect` со scroll-handler'ом.

## Файлы

- [modules/ui_web/src/pages/DialogsPage.tsx:115-129](../../modules/ui_web/src/pages/DialogsPage.tsx#L115-L129) — заменил `<ScrollArea>` на обычный `<div className="overflow-y-auto">`, внутри `flex-col-reverse`.

## Проблема

Backend (`/dialogs/{id}/messages?direction=backward` по дефолту) возвращает сообщения `ORDER BY date DESC` — новые первыми. UI мапил прямо: первый элемент массива попадал в первую строку списка, пользователь читал диалог снизу-вверх, как антимессенджер.

На скриншоте диалога было именно так:
```
13:33  "короче вот такой додик"        ← самое свежее, сверху
13:29  "а у тебя как по финансам?"
13:28  "пару тачил"
...
13:28  "а наследство прям солидное?"   ← более старое, снизу
```

В Telegram / WhatsApp / iMessage — наоборот.

## Решение

### Вариант А (который выбрал) — `flex-col-reverse`

```tsx
<div className="overflow-y-auto">
  <div className="flex flex-col-reverse gap-2">
    {items.map(m => <MessageBubble key={m.id} m={m}/>)}
  </div>
</div>
```

DOM-порядок детей совпадает с порядком массива (`items[0]` = первый child). `flex-col-reverse` меняет **визуальный** порядок: первый child рендерится в нижнем конце flex-контейнера.

Итог:
- `items[0]` = самое свежее → внизу.
- `items[items.length-1]` = самое старое → вверху.
- Scroll по умолчанию `scrollTop=0` — для reverse-контейнера это физически низ = где самое свежее. Залогинились в диалог — сразу видим последнее сообщение, как в любом мессенджере. **Без `useEffect` и `scrollTo`.**

### Вариант Б — развернуть массив клиентом `[...items].reverse().map(...)`

Работает, но:
- Теряем халявный «скролл внизу» — нужно либо `ref` + `useEffect` + `scrollIntoView`, либо ручной менеджмент `scrollTop`.
- При пагинации (`has_more=true`, подгрузка старых через скролл вверх) — нужно корректно сохранить scroll offset, иначе страница прыгает. С `flex-col-reverse` эта проблема отдельно обрабатывается браузером: при префиксе новых элементов в конец массива (новые снизу) прокрутка не дрожит.

### Вариант В — запрашивать у backend ASC (`direction=forward`)

Не работает для initial-load: надо знать курсор откуда начинать. Для UI, где открываешь диалог и видишь последние 50 — `backward` естественно.

## Почему убрал Radix ScrollArea

`<ScrollArea>` оборачивает в Viewport со своими перехватами скролла. `flex-col-reverse` в нём вёл себя странно: Viewport ставит transforms, а reverse внутри этого выглядит как зеркальное отображение. Простой `<div className="overflow-y-auto">` работает предсказуемо и без костылей. Плашка скроллбара получается дефолтная (браузерная) — для dev-панели некритично, куча браузеров уже прячут её когда не нужна.

ScrollArea всё ещё используется в левой колонке списка диалогов — там он ценен (узкий кастомный бар).

## Проверка

Открыть диалог:
- Сверху в ленте — самое старое сообщение из загруженных (из последних 50).
- Внизу — свежее.
- При открытии диалога курсор/scroll уже в самом низу, не нужно скроллить.

## Открытые вопросы

- **Infinite scroll вверх.** `messagesQ.data.next_cursor` есть, но я его пока не использую — при скролле к верху нужно подгружать следующие 50 старых. Для этого:
  1. Привязать `IntersectionObserver` к «sentinel»-div'у в начале списка (в DOM = конец массива из-за reverse).
  2. Когда он попадает в viewport, вызвать `useInfiniteQuery` со следующим курсором.
  3. `tanstack/react-query` умеет `useInfiniteQuery` и сам склеит страницы — переделать `useMessages` под него. Сделаю отдельным этапом когда станет нужно (сейчас 50 сообщений на открытие хватает).
- **Автоматический скролл к новому входящему через SSE.** Когда `/dialogs/{id}/stream` будет wire'нуто и прилетит `message.saved` — хорошо бы авто-скроллить, но только если пользователь **уже внизу**. Если он прокрутил в историю, новые должны приходить тихо. Проверка «user-at-bottom» (`scrollTop + clientHeight >= scrollHeight - 10`) — стандартный приём, сделаем вместе со стримом.
- **Prev scroll position при пагинации.** flex-col-reverse помогает с префиксом массива снизу, но DevTools для проверки не гонял. Если при загрузке старых вверх ленту будет «дёргать», придётся явно сохранять `scrollHeight` до и после и подправлять scrollTop.
