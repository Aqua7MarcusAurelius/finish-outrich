# 8.4 — Диалоги: фиксированная ширина колонки сообщений

**Дата:** 2026-04-24
**Этап:** 8.4
**Коммит:** `1184ebf`

## Что сделано

Обернул ленту сообщений и шапку диалога в `max-w-[780px] mx-auto`. Пузыри сообщений больше не разъезжаются на полный экран — компактная колонка по центру, пустое место по бокам.

## Файлы

- [modules/ui_web/src/pages/DialogsPage.tsx:98-117](../../modules/ui_web/src/pages/DialogsPage.tsx#L98-L117)
  — `<div className="mx-auto max-w-[780px] …">` в двух местах: шапка диалога и контейнер ленты.

## Проблема

На мониторе 1920px после левой колонки (~260px) и навигации правая область ≈1600px. `MessageBubble` имел `max-w-[78%]`, выравнивался `justify-start/end` по is_outgoing. Формально всё работало, но визуально:

- Входящее «пару тачил» — левый край экрана.
- Исходящее через строку «а у тебя как по финансам?» — правый край.
- Между ними 1000+ пикселей пустоты.
- Глаз прыгает на 60см туда-сюда, терялся контекст диалога.

Скриншот пользователя: ровно эта картина.

## Решение

Одна обёртка `mx-auto max-w-[780px]` вокруг:
- контейнера шапки диалога (имя, @username, dialog #N);
- контейнера ленты сообщений.

Внутри — всё остаётся как было: `justify-start/end`, `max-w-[78%]` у bubble, цвета входящих/исходящих. Просто «холст» стал узким.

Ширина 780px — из проб:
- 600px — слишком узко, длинные сообщения обрезались в слишком много строк.
- 1000+ — проблема не решается.
- 720–800px — размер колонки Telegram Desktop, ощущается привычно.

## Почему так, а не иначе

### Почему не убрать left/right вообще и выстроить всё в одну колонку

Вариант был — как в IRC / Slack, где все сообщения слева, а стиль исходящих отличается цветом/префиксом. Не взял:

- Проект режимит социальный мессенджер Telegram, пользователь привык к разделению left/right.
- Теряем одну точку быстрого восприятия «кто писал».
- При сужении колонки до 780px визуальный gap между бортами становится нормальным — как в самом Telegram.

### Почему 780px, не `max-w-2xl` / `max-w-3xl`

Tailwind `max-w-2xl` = 42rem = 672px, `max-w-3xl` = 48rem = 768px. Близко, но хотелось чуть больше под русский текст (кириллица шире). `780px` — одно конкретное число, понятно откуда. Можно было бы вытащить в CSS var `--dialog-column`, но пока одно место использования.

### Почему `mx-auto` по центру, не прижать к левому краю колонки

Соблазн: сэкономить правое пустое поле, прижав ленту влево. Взвесил против:

- Когда правое поле равно 0, следующее пустое пространство — ресайз окна вниз. Ощущается «поломанно».
- По центру — симметрично, глаз не искажает фокус.
- Пустые поля слева и справа зеркально используем под будущие виджеты (профиль собеседника справа, список связанных событий шины слева).

### Почему не ограничил ширину `MessageBubble` абсолютом вроде `max-w-[500px]`

Вариант: оставить правую панель широкой, а каждому пузырю жёстко задать max-width. Работает, но:

- Исходящие всё равно упираются в правый край, входящие в левый. Сильный gap.
- Нельзя использовать пустоту под виджеты — центр пуст, края заняты пузырями.

Узкая колонка решает обе проблемы разом.

## Свободное место под будущие виджеты

Пустые поля слева и справа от колонки сообщений (по ~400px на каждый при FullHD) открыто под:

- **Справа** — профиль собеседника: `dialogs.bio`, `birthday`, дата контакта, внутренний `dialog_id`, кнопки «открыть в event log по этому диалогу», «retranscribe/redescribe по media в диалоге».
- **Слева** — связанные события шины: последние `transcription.*` / `description.*` / `message.*` с `data.dialog_id == current`, с подсветкой ошибок. Это, по сути, микро-версия event log под конкретный диалог.

Делать их прямо сейчас не стал — они требуют отдельных API-интеграций, а вопрос пользователя был про читаемость. Читаемость закрыта одной оберткой.

## Проверка

Открыть http://localhost:3000/#/dialogs/1/95 и убедиться:
- Имя собеседника и «@vians777 · dialog #95» сдвинулись от левого края к центру, теперь по центру над колонкой.
- Входящее «пару тачил» и исходящее «короче вот такой додик» разделены ~170px gap'ом, не 1000px.
- При ресайзе окна ширина колонки фиксирована на 780px, пустые поля симметрично увеличиваются/уменьшаются.

## Открытые вопросы

- **Виджеты по бокам.** Дизайн и реализация — отдельным этапом (см. выше).
- **Адаптив под малые экраны.** На 1366×768 пустого места по бокам меньше, на 1024px ширины почти нет — всё «как всегда». Ниже 780px колонка сжимается благодаря `max-w` — ломаться не должна, но стоит проверить. Проверял только FullHD.
- **Width в preference'ах.** Можно было бы дать пользователю выбирать узкую/среднюю/широкую колонку. Пока не нужно — один режим по центру.
