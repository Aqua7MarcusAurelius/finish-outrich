# 6.2 — ffmpeg: нарезка кадров из видео/GIF/video_note

**Дата:** 2026-04-22
**Этап:** 6 (модуль описания медиа)

## Что сделано

Новый модуль `modules/media_description/ffmpeg.py` с функцией
`extract_frames(input_bytes, count) -> list[bytes]`. Возвращает N
JPEG-кадров, равномерно распределённых по длительности файла.

## Файлы

- `modules/media_description/__init__.py` — пустой пакетный init.
- `modules/media_description/ffmpeg.py` — `extract_frames`,
  `FfmpegError`, `NoFramesError`.

## Почему так, а не иначе

### Временный файл, а не stdin

Ровно та же проблема, что при транскрипции (см. запись этапа 5,
`STAGE_5_REPORT.md`, fix 2): Telegram mp4/video_note хранит moov atom
в конце файла, ffmpeg через stdin seek'ать не умеет — видеодорожку не
находит. С файлом на диске работает для любого формата. Тот же приём
с `tempfile.mkstemp` + `asyncio.to_thread` для I/O вне event-loop'а.

### ffprobe → duration, отдельные запуски ffmpeg за кадр

Альтернатива — один запуск ffmpeg с фильтром `-vf fps=N/duration` или
`select='eq(n,...)'`. Отвергнуто: фильтры хрупкие (зависят от знания
fps и длительности, а для vfr-видео и GIF ведут себя непредсказуемо),
запуск с `-ss` и `-frames:v 1` — тупо и контролируемо. Для 3-5 кадров
оверхед на запуск процесса — несколько десятков мс, не проблема.

### JPEG, не PNG

Vision-модели принимают и то и другое, но JPEG базируется в base64
обычно в 5-10 раз меньше по байтам. В payload через HTTP это разница
в секундах отправки и в токенах у провайдера. Визуальное качество
для «опиши что изображено» — избыточно, хватает `-q:v 3`.

### Моменты: середины равных отрезков, а не 0..1

Формула `(i + 0.5) * duration / count` даёт `0.5/N, 1.5/N, ..., (N-0.5)/N`.
Причины:
  - В кадр 0 часто попадает чёрный экран или лого.
  - В самый конец — тоже часто чёрный/статичный кадр.
  - Середины равных отрезков дают равномерное покрытие без краевых
    эффектов.

Зажим `min(m, duration - 0.05)` — защита от запроса кадра ровно на
конец (ffmpeg может вернуть пустой результат).

### Порог MIN_FRAME_SIZE = 500 байт

JPEG даже с максимальным сжатием и маленьким разрешением — сотни
байт заголовка + данные. 500 байт — заведомо битый/пустой. Такие
отсеиваем молча; если все N получились битыми — `NoFramesError`,
наверху трактуется как «нечего описывать».

### `-ss` до `-i` (быстрый seek)

Точность ~ближайший ключевой кадр, что для описания содержимого
более чем достаточно. `-ss` после `-i` был бы точнее, но декодировал
бы всё видео от начала до момента — медленно на длинных файлах.

## Открытые вопросы

- **GIF без явной длительности в контейнере.** Некоторые GIF не
  отдают `format.duration` через ffprobe. На такой упадём в
  `NoFramesError` → `description.done [failed, no_frames]`.
  Если окажется что это частый кейс, можно дополнительно считать
  `nb_frames` и делить на fps. Пока оставлено как есть.
- **Длительность 0 для статичных изображений (если вдруг придут
  сюда).** Мы фильтруем по типу в сервисе (6.3), сюда попадут только
  `gif/video/video_note`. Для подстраховки `duration <= 0 →
  NoFramesError` — предсказуемая ошибка, а не падение.
