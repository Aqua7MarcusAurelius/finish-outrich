## Фикс 1 — read-receipts при `in_chat=True`

### Проблема

`_mark_dialog_read_safe` вызывался только один раз — в `_enter_after`
при переходе `InChat=0 → 1`. Пока сессия уже в чате и прилетает новое
inbound — сообщение у собеседника оставалось без галочек прочтения до
выхода по idle (3 мин) и следующего enter'а. Это выдавало бота:
первое сообщение «прочитано», следующие — нет.

### Решение

В `_on_inbound`, в ветке `else` (когда `self.in_chat == True`), один
дополнительный `await`:

```python
else:
    # InChat=1: живой человек в открытом чате сразу видит сообщение.
    # Помечаем весь диалог прочитанным (read_message без message_id
    # делает ReadHistoryRequest до последнего).
    await self._mark_dialog_read_safe()
    self._bump_planner()
    self._reset_idle()
```

`wrapper.read_message(tg_user_id)` без `message` аргумента → Telethon
делает `send_read_acknowledge(entity, message=None)` → `ReadHistoryRequest`
с `max_id = last_message_id` → весь диалог помечен прочитанным. Именно
то поведение живого человека с открытым чатом.

### Что НЕ меняется

- `in_chat=False` ветка — прежняя, `_mark_dialog_read_safe` сработает
  один раз при переходе в чат.
- Ошибки чтения глушатся в `_mark_dialog_read_safe` (try/except) —
  read-receipt не критичная операция.

## Фикс 2 — restore active sessions при старте AutoChatService

### Проблема

В MVP per-session task не переживал рестарт приложения. Строка в БД
оставалась `status='active'`, но in-memory индексы `_sessions_by_id`,
`_by_tg_user`, `_by_dialog` — пустые. Входящие события молча дропались
диспатчером (`session is None → return`). Лечилось только ручным
`stop_session` + `create_session` с повторной генерацией initial.

### Решение

Новый метод `AutoChatService._restore_active_sessions()`, вызывается
в начале `run()` после `ensure_group` и до consumer-loop'а:

```python
async def run(self) -> None:
    await bus.ensure_group(AUTOCHAT_GROUP)
    await self._restore_active_sessions()
    log.info("AutoChatService loop started")
    while not self._stop_event.is_set():
        ...
```

Для каждой строки `WHERE status='active'`:
1. `data = dict(row); data["in_chat"] = False` — принудительный сброс
   `in_chat` (таймеры enter/idle не переживают рестарт; следующее
   inbound нормально запустит enter-timer, как описано в
   [docs/autochat.md:425](../autochat.md#L425)).
2. Создаём `AutoChatSession` из обновлённой строки.
3. Регистрируем в `_sessions_by_id`, `_by_tg_user`, и если `dialog_id`
   есть — в `_by_dialog`.
4. `await session.start()` — три asyncio-таска встают.

### Тонкости

- **Воркер может быть ещё не запущен** (в проекте воркеры стартуются
  вручную через `POST /workers/{id}/start`, не авто). Это ок: сессия
  просто ждёт событий, а когда пользователь поднимет воркер и в TG
  прилетит новое сообщение — state machine оживёт. Ошибка в sender'е
  при отсутствии wrapper'а приведёт к `_fail` — но до её материализации
  нужен inbound, которого без воркера не будет.
- **Persist in_chat=false в БД**: выполняется автоматически первым
  `_persist_state()` внутри `session.start()` — отдельный UPDATE не
  нужен.
- **Порядок в lifespan**: AutoChatService создаётся после WorkerManager.
  WorkerManager.__init__ не поднимает воркеры сам, так что на момент
  restore воркеры ещё могут быть вне памяти — не страшно по пункту
  выше.

### Что осталось за скоупом

Не добавляли отдельного `autoreply_enabled bool` на уровне dialog'а
или session'а. Текущий `status` enum (`starting/active/paused/failed/
stopped`) полностью покрывает use-case «выключен / включён» — нужен
будет pause/resume, добавим позже с одним endpoint'ом, без миграций.

## Фикс 3 — timestamp'ы в role-based истории

### Проблема

В role-based режиме (`{conversation_history}` НЕ присутствует в
`autochat_reply_system.md`) история отдавалась LLM как
`{role: user/assistant, content: body}` — без даты-времени сообщения.
В system подставлялся только `{current_time}`. Opus видел ЧТО сказали,
но не КОГДА.

Реальный пример из диалога 45: собеседник после 19-часовой паузы
написал «Да, привет как ты, вчера был занят». Без timestamp'а LLM не
мог понять, что имеется в виду именно вчера — предыдущее наше сообщение
в контексте могло быть «минуту назад».

### Решение

В [generation.py:302-310](../../modules/autochat/generation.py#L302-L310)
префиксуем content каждого turn'а:

```python
for t in turns:
    role = "assistant" if t["is_outgoing"] else "user"
    when = t["date"]
    if isinstance(when, datetime):
        stamp = when.strftime("%Y-%m-%d %H:%M")
        content = f"[{stamp}] {t['body']}"
    else:
        content = t["body"]
    out.append({"role": role, "content": content})
```

Формат `[%Y-%m-%d %H:%M]` — тот же что в `_format_history_text` для
conversation_history-режима. UTC, без таймзоны.

### Как это выглядит для Opus

```
{"role": "assistant", "content": "[2026-04-22 17:15] привет) разгребаю рабочую телегу..."}
{"role": "user",      "content": "[2026-04-23 12:11] Да, привет как ты, вчера был занят"}
{"role": "user",      "content": "[2026-04-23 12:11] не успел отписать"}
{"role": "assistant", "content": "[2026-04-23 12:14] мм, подожди, ты сейчас как будто..."}
```

Opus теперь может опираться на реальный временной контекст — и различать
«только что» от «вчера», «пару часов назад» от «пару минут назад».

### Что НЕ меняется

- Медиа-пометки внутри body остаются как были — `[голос 5с: «…»]`
  просто оказывается после `[timestamp] `.
- Многострочные сообщения: timestamp впереди всего тела, внутренние
  переносы строк сохраняются. Отдельного timestamp'а на каждую строку
  не делаем — зашумило бы без пользы.
- Таймзона — UTC. Если потребуется МСК, правится в двух местах:
  [generation.py:277](../../modules/autochat/generation.py#L277) и
  [generation.py:305](../../modules/autochat/generation.py#L305).
